## はじめに

「Claude Codeに実装を丸投げしたら2週間でアプリが完成した」——そう言うと聞こえはいいですが、正確ではありません。実際に起きたのは、**開発のボトルネックが「コードを書く時間」から「決める時間」に移った**ことです。

減量目標(10月末までに-8kg)のために、体重・食事・水分・筋トレ・日記を記録して週次レビュー+AIコーチングで振り返るローカルファーストPWA「からだログ」を個人開発しています。要件定義を書き始めたのが7月1日、実装開始が7月5日。約2週間の実績はこうです。

- Issue 77件、PR 60本(squashマージ運用)
- 自動テスト370ケース
- 記録系のMVPからAIコーチング・Garmin連携・血圧/習慣トラッカーまでの4フェーズが実装済み
- 実装の大部分はClaude Codeが書き、人間(私)はほぼ仕様の決定とレビューだけ

このボトルネックの移動を前提にすると、人間側の仕事は「決めたことを、AIが迷わない形で残す」ことになります。この記事では、そのために使っているドキュメント構成と運用ルールを、実際のリポジトリから紹介します。

## AIエージェント開発で実際に起きる問題

Claude Codeのようなエージェントにある程度の規模の開発を任せると、コーディング能力とは別の問題に突き当たります。

**セッションは記憶を持たない。** 今日のセッションは、昨日のセッションで交わした「同期先は既存シートじゃなくて新規シートにしよう」という会話を知りません。何も対策しなければ、新しいセッションは過去の決定を平気で蒸し返し、矛盾する実装を生みます。

**「なぜ」が失われる。** コードには「何をしているか」しか残りません。「グラフの目標線がなぜ黄色ではなくグレーなのか」(デザインガイドで黄色は達成演出専用と決めたから)という理由を知らないAIは、良かれと思って目標線を目立つ黄色に「改善」します。

**暗黙の不変条件を踏み抜く。** 「`synced: true` を直接セットしていいのは同期エンジンだけ」のようなルールは型には現れないので、grepで似たコードを見つけたAIが悪気なく違反します。

どれもAIが馬鹿だから起きるのではなく、**人間の新規メンバーがオンボーディングなしで参加したら起きること**がそのまま起きているだけです。なので対策も同じで、オンボーディング資料を整備することになります。違いは、AIは何度でも文句を言わずに読み直してくれることです。

## 全体像: ドキュメントの役割分担

リポジトリのドキュメントはこう分かれています。

```
CLAUDE.md                          … AIの案内図(コマンド、開発フロー、アーキテクチャの要点、禁止事項)
docs/
  からだログ_要件定義書.md          … 何を作るか・なぜ作るか・何を作らないか
  からだログ_画面設計書.md          … 画面仕様とデータモデル、同期フロー
  からだログ_デザインガイド.md      … パレット、タイポグラフィ、レイアウトルール
  からだログ_AIコンサルティング設計書.md … AIコーチングのデータ契約・プロンプト・プライバシー
  からだログ_意思決定ログ.md        … 決定済み論点の「経緯」の記録
  からだログ_レビューチェックリスト.md … 全体レビューの機械的な手順書
GitHub Issues                      … 残作業の唯一の管理場所(バックログのmdファイルは作らない)
```

ポイントは情報の置き場所を役割で厳密に分けていることです。

| 知りたいこと | 見る場所 |
|---|---|
| 現在の仕様 | 仕様書本文(要件定義書・画面設計書) |
| その仕様に至った経緯・捨てた代替案 | 意思決定ログ |
| 何が残っているか | GitHub Issues |
| コードの触り方・落とし穴 | CLAUDE.md |

この分離はAIのためだけではなく、情報の重複と矛盾を防ぐためでもあります。同じ事実を2か所に書くと必ず片方が古くなるので、「真実の情報源はどれか」を各ドキュメントの冒頭に明記しています(CLAUDE.mdにも「真実の情報源はこれらの仕様書であり、このファイルではない」と書いてある)。

## 仕組み1: CLAUDE.md には「地図」と「地雷」だけを書く

CLAUDE.mdはAIが毎セッション最初に読むファイルです。ここに仕様の詳細を書きたくなりますが、書くと仕様書と二重管理になります。代わりに書いているのは次の3種類です。

**1. コマンドと環境の事実。** `npm run test` の実行方法、そして「`npm run lint` はpackage.jsonに定義されているがESLintは実際には入っていない — 当てにしないこと」のような、試すと時間を溶かす罠。

**2. 型からは読み取れないアーキテクチャ上の選択。** 例えばこのアプリでは、体重記録は日付(YYYY-MM-DD)を主キーにして「1日1件・後勝ち」を `put()` だけで成立させ、食事記録はUUIDキーで同日複数件を許しています。この非対称は型定義を眺めても意図が分からないので、理由ごとCLAUDE.mdに書いています。

**3. 禁止事項を、必ず理由付きで。** 例:

> `getUnsyncedWeightRecords` はDexieのインデックスではなくJS側の `.filter()` で絞り込んでいる — IndexedDBはbooleanをインデックスのキーにできないことと、この規模では十分軽いため。**これをインデックス化して「最適化」しないこと。**

理由なしの禁止は、賢いAIほど「古い制約だろう」と判断して破ります(人間の新人と同じです)。「なぜダメか」までセットで書くと、ほぼ確実に守られるようになりました。`public/_redirects` を追加するとCloudflareのデプロイが無限リダイレクトで落ちる、`useLiveQuery` は「ロード中」と「undefinedに解決」を区別できないので `?? null` で正規化する、といった一度踏んだ地雷は、踏むたびにここへ追記しています。**CLAUDE.mdは書いて終わりではなく、事故のたびに育てる運用ファイル**です。

## 仕組み2: 意思決定ログ — 「経緯」を仕様書から追い出す

最初は要件定義書の中に「検討の経緯」を書いていましたが、決定が増えるほど本文が読みにくくなるので、**経緯だけを意思決定ログに分離**しました。書式はシンプルで、テーマ別の節に日付付きで追記していくだけです。

> **2026-07-11: マルチユーザー化の方針検討(#67) → 費用対効果が見合わないと判断し対応不要としてクローズ。** (…検討した3形態と、見送った理由が続く…)

これが効くのは、AIが「やらないと決めたこと」を提案し直さなくなることです。機能追加を頼むと、AIはまず意思決定ログを確認し、「クラウドDBは#67で見送り済みなので、既存のシート同期に乗せます」という前提から始めてくれます。過去の議論の再演がなくなるのは、体感でかなり大きい時間の節約です。

注意点として、ログはあくまで経緯の記録で、**現在の仕様の正は仕様書本文**というルールを崩さないこと。「ログに書いたから仕様書は更新しなくていい」を許すと真実の情報源が2つに割れます。

## 仕組み3: Issue駆動 + 仕様書への書き戻しを「必須」にする

日々の開発ループはこうです。

1. **非自明な作業は着手前にIssueを立てる**(1人開発でも)。Issueには元になった仕様書の該当章をリンクする
2. mainを最新化してブランチを切り、実装。`npm run test` / `npm run build` で検証、UI変更はPlaywrightでスクリーンショット確認
3. **仕様に影響する変更は、該当する仕様書本文を現在の仕様に書き換える(必須)。** 見送っていた論点を実装したら、論点リストと意思決定ログにも経緯を残す
4. PR作成前にレビューチェックリストの観点で自己点検
5. PRを作成し(`Closes #N`)、ユーザーの明示的な指示でsquashマージ

このループ自体をCLAUDE.mdの「開発フロー」節に書いてあるので、AIは指示されなくてもこの手順で動きます。特に3の**書き戻しの必須化**が仕様書駆動の生命線です。書き戻しをサボると仕様書は数日で「かつての構想メモ」に劣化し、AIがそれを真実として読む事故が始まります。逆に書き戻しが回っている限り、新しいセッションはいつ来ても最新の仕様から出発できます。

「純粋なバグ修正・リファクタは書き戻し不要。ただし影響の有無は毎回判断する」という逃げ道も明文化しています。全部書き戻せにすると運用が破綻するので、判断基準ごとルール化するのがコツです。

## 仕組み4: レビューを「チェックリスト」として手順化する

一度リポジトリ全体レビューをやった際、その観点を使い捨てにせず `docs/からだログ_レビューチェックリスト.md` に手順化しました。ポイントは**判断の少ない機械的な手順**に落とすことです。例えばコード不変条件のチェックは、grepコマンドと合格条件のセットになっています。

| チェック | コマンド | 合格条件 |
|---|---|---|
| `synced: true` の直接セット | `grep -rn "synced: true" src/` | ヒットは同期エンジン・取り込み・テストのみ |
| 絵文字の混入 | `grep -rnP "[絵文字レンジ]" src/` | 0件(アイコンはSVGコンポーネントのみ) |
| accent色の誤用 | `grep -rn "FFC145" src/` | 「達成の瞬間の演出」のみ |
| WorkerのローカルTZ依存 | `grep -n "getHours..." worker/` | 0件(WorkerはUTC動作) |

ここまで機械化しておくと、「全体レビューして」の一言で、どのセッションのAIでも同じ品質の点検が再現されます。**レビュー観点の属人化(属セッション化?)を防ぐドキュメント**という位置づけで、これも事故のたびに項目を足しています。実際、無認証APIから健康データが読めるという本物の指摘は、この全体レビューから出てきました。

## 2週間やってみての実感

**効いたこと:**

- **決定の再利用がきく。** 似た機能(血圧・周囲径・習慣トラッカー)の追加が「既存パターンに乗せて」の一言で済む。パターン自体がドキュメント化されているため
- **セッションをまたぐ手戻りがほぼ消えた。** 過去の決定と矛盾する実装が出てこない
- **人間の仕事が明確になる。** 私がやるのは、論点に決着をつけること・モックや実画面を見て判断すること・マージの承認。コードレビューはするが、書くことはほぼない

**コストと限界:**

- ドキュメント維持は無料ではない。書き戻し込みで、体感では作業時間の2〜3割がドキュメント
- それでもズレは発生する。だからレビューチェックリスト(3章の整合チェック)が保険として要る
- 仕様書に書いていない暗黙の期待は、当然実装されない。「書いていないものは存在しない」と思って書く必要がある

このコストは、1回きりの使い捨てコードなら明らかに払い過ぎです。数週間以上続くプロジェクトで、セッション(または人)をまたいで文脈を引き継ぐ必要があるなら、十分に元が取れる — というのが現時点の結論です。

## まとめ

- AIエージェントとの開発で起きる問題は、オンボーディングなしの新規メンバーに起きる問題と同じ。対策もオンボーディング資料の整備
- 情報は役割で置き場所を分ける: 現在の仕様=仕様書、経緯=意思決定ログ、残作業=Issue、触り方と地雷=CLAUDE.md
- 禁止事項は理由付きで書く。理由のない禁止はAIに破られる
- 実装後の仕様書への書き戻しを必須ルールにする。これが崩れると全部崩れる
- レビュー観点はチェックリストとして手順化し、どのセッションでも再現可能にする

このアプリ自体(実測TDEEの逆算、AIには計算をさせず言語化だけさせる設計、Google Sheetsを個人用DBにする同期エンジン)の話は、それぞれ別記事で書く予定です。減量実験の経過はnoteで週次レポートとして公開していきます。
