## はじめに

減量記録PWA「からだログ」には、体重・カロリー・水分・歩数・睡眠・血圧など10本近い推移グラフがあります。チャートライブラリは使っていません。全部素のSVGをReactコンポーネントとして手書きしています。

「ライブラリを使わない」は普通は悪手ですが、要件次第では手書きの方が総コストが安くなります。この記事では、実際のコード(1グラフ約150行)を見せながら、どういう要件なら手書きが勝つのかを書きます。コードは公開しています: https://github.com/yotti773/lifelog

## なぜライブラリを入れなかったか

理由は「ライブラリのデフォルトと戦いたくなかった」からです。

このアプリにはデザインガイドがあり、色にはそれぞれ役割が決まっています。中でも特殊なのがアクセントの黄色で、**「達成の瞬間」の演出(目標達成・連続記録)専用に予約**されていて、常時表示される静的UIに使ってはいけないルールです。だからグラフの目標線は目立つ黄色ではなく、ミュートグレーの破線です。

この種の「うちのデザインシステムの都合」をチャートライブラリで貫くには、テーマAPIを調べ、上書きできない部分をCSSで殴り、バージョンアップのたびに直す作業が発生します。一方、要件を数えてみると:

- 折れ線と棒グラフだけ。ズーム・パン・ツールチップ・アニメーション・凡例は要らない
- データ点は多くて数百(1日1点×数ヶ月)
- モバイル縦持ち固定で、サイズはカード幅いっぱいの1パターン

インタラクティブ性が要らない静的な折れ線・棒グラフは、SVGで書くと本当に小さいのです。ライブラリの学習・格闘コストの方が確実に高い、と判断しました。バンドルが数百KB増えないのはPWAとしてのおまけです。

## 折れ線グラフの中身は3つの式だけ

体重グラフの本体を抜き出します。やっていることは「スケール関数を作る → pathの`d`文字列を組み立てる」だけです。

```tsx
const WIDTH = 320;
const HEIGHT = 180;
const PADDING = { top: 18, right: 12, bottom: 24, left: 34 };

// X: 日付(時刻ms)→ 横位置
const xScale = (time: number) =>
  PADDING.left + ((time - minTime) / timeSpan) * plotWidth;

// Y: 体重 → 縦位置(SVGは下が正なので反転)
const yScale = (weight: number) =>
  PADDING.top + plotHeight - ((weight - (minWeight - yPad)) / scaledSpan) * plotHeight;

// 折れ線: M x,y L x,y L x,y ...
const linePath = points
  .map((p, i) => `${i === 0 ? "M" : "L"}${p.x.toFixed(1)},${p.y.toFixed(1)}`)
  .join(" ");
```

描画は`viewBox="0 0 320 180"`のSVGに、この`linePath`と目盛り・点を置くだけです。`viewBox`+`width: 100%`にしておけば、レスポンシブ対応はブラウザのSVGスケーリングが勝手にやってくれます。座標計算はすべて320×180の固定論理座標で書けるので、画面幅のことは一切考えません。

D3を部分的に使う手(スケールだけ`d3-scale`)もありますが、線形スケール2本のためだけに依存を足す量ではない、というのが上のコードの通りです。

## 手書きで一番難しいのは「目盛り」

線を引くのは簡単で、手書きグラフの本当の敵は軸の目盛りです。「71.234〜73.891の範囲を、人間が読みやすい切りのいい値で3本くらいに区切る」を自前で書く必要があります。

これはよく知られた「1・2・5×10^n刻み」のアルゴリズムで解けます。

```typescript
export function axisTicks(min: number, max: number, targetCount = 3): AxisTick[] {
  const rawStep = (max - min) / targetCount;
  const magnitude = 10 ** Math.floor(Math.log10(rawStep));
  const step = [1, 2, 5, 10].map((m) => m * magnitude).find((s) => s >= rawStep) ?? 10 * magnitude;

  const ticks: AxisTick[] = [];
  // 加算による浮動小数の蓄積誤差(0.30000000000000004など)を各値で丸め直す
  for (let value = Math.ceil(min / step) * step; value <= max + step * 1e-6; value += step) {
    const rounded = Number(value.toFixed(10));
    ticks.push({ value: rounded, label: step < 1 ? rounded.toFixed(1) : rounded.toLocaleString() });
  }
  return ticks;
}
```

55行のこのファイル(横軸の間引きロジック込み)を全グラフで共有していて、単体テストもここに集中しています。ライブラリを捨てる決断は、実質「このファイルを自分でメンテする」決断です。逆に言えば、目盛り計算さえ部品化すれば残りは単純な座標計算しかありません。

浮動小数の`for`ループ蓄積誤差(`0.1+0.2`問題)でラベルが`70.30000000000000004`になる、SVGは上が0なのでY軸を反転し忘れる、あたりが実際に踏んだ小さい罠です。

## ライブラリでは面倒な「うちの仕様」がタダで入る

手書きの利点が一番出たのは、デフォルト挙動との戦いが消えたことです。例えば:

- **縦軸レンジは実測値だけから決める。** 目標体重をレンジに含めると、目標が遠い間は実測の変動が上端に圧縮されて傾向が読めなくなります。だから目標線は「算出済みレンジ内に収まるときだけ」描き、範囲外なら右上のラベルだけで示す — この条件分岐は自分のコードなので3行です
- **欠測は隙間として見せる。** 記録が無い日を線で補間してつなぐと、サボった事実が隠れます。体重は欠測日に点を打たず、カロリーはデータ層で全日0埋めして空白の棒を見せる、と指標ごとに欠測の表現を選んでいます
- **最新点だけ強調する。** 「今どこにいるか」が主役なので、最後の点だけ大きくリング付きで描く。ライブラリだとカスタムレンダラーを書く話が、手書きなら`<circle>`を2つ足すだけ

どれも大した処理ではありません。ただ、ライブラリの上でやろうとすると「その設定項目があるか探す→無ければ拡張ポイントと戦う」になる類のものです。**要件が「描画のカスタマイズ」に寄っているほど、手書きの相対コストは下がります。**

## 手書きにしない方がいい場合

公平のために、逆条件も書いておきます。

- **ツールチップ・ズーム・ブラシ選択などのインタラクションが要る** — ここからが本当に大変な領域で、ライブラリの価値はほぼここにあります
- **チャート種類が多い**(散布図・箱ひげ・ローソク足…) — 種類ごとに座標計算を書くのは割に合いません
- **軸が複雑**(対数軸・時間軸の自動目盛り・多軸) — 目盛り問題の難易度が跳ね上がります
- **デザインの独自性が要らない** — デフォルトで満足できるなら、そもそも戦いが発生しません

このアプリは「静的・少種類・強いデザイン制約」の三拍子が揃っていたから手書きが勝った、という話であって、汎用の推奨ではありません。

## まとめ

- 静的な折れ線・棒グラフの本体は、スケール関数2本とpath文字列の組み立てだけ。`viewBox`でレスポンシブもタダ
- 手書きの本丸は目盛り計算。「1・2・5刻み」アルゴリズムを共有部品+単体テストにする
- デザインシステムの制約が強い・欠測の見せ方にこだわるなど、カスタマイズ要件が濃いほど手書きが有利
- インタラクションが要るなら素直にライブラリへ。この判断は要件を数えてから
