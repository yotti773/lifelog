# からだログ 意思決定ログ

作成日: 2026-07-06
関連ドキュメント: からだログ_要件定義書.md、からだログ_画面設計書.md

要件定義書・画面設計書に混在していた「決定済み論点(経緯の記録)」を本ログに集約する。**本ログは経緯の記録であり、現在の仕様の真実の情報源は要件定義書・画面設計書の本文**である(本文と食い違う場合は本文が正)。新しい決定は末尾ではなく該当テーマの節に追記し、日付が分かるものは日付を添える。

## 全体方針・フェーズ

- **2026-07-06: アプリの最終形を「必要なことを入力するとパーソナルなコンサルティングができるアプリ」と再定義した(フェーズ3)。** 記録(現状把握)だけでなく「記録 → 評価 → 助言 → 行動修正」のループを回すことを長期目標とする。フェーズ3の機能構成は以下で決定(要件定義書4.7章):
  - 採用: 身体プロフィール+目標カロリー自動計算(#43)、実測TDEE推定(#44)、週次レビュー(#45)、記録率の可視化(#46)、PFC目標値(#47)、AIコーチング(#12)
  - 見送り: 活動データ(歩数・運動消費)の手入力欄の先行実装 → 実測TDEEの逆算(#44)があれば活動量データが無くても収支評価は成立するため、Garmin/Google Fit連携(#11・#13)に統合し単独では実装しない
- **2026-07-06: AIコーチングの実現方式** → 数値の計算・集計・安全判定はすべてコード側で決定論的に行い(`WeeklyDigest`)、AIには解釈と言語化だけをさせる構成で決定。モデルは低コストな軽量モデル(Gemini Flash-Lite級)を既存のWorker中継で呼び出す。詳細はAIコンサルティング設計書を参照
- **2026-07-12: 週次レビューへの筋トレ・水分・日記の統合(#103) → 3つとも統合することに決定・実装済み。** 筋トレ(記録日数・種目数・総セット数)・水分(記録がある日の平均摂取量・目標達成日数)は`WeeklyDigest`の週サマリーとして追加し、画面表示とAIコーチングの入力の両方に使う(活動(#82)と同じ「記録が無い週は省略」方式)。日記は週内の本文・気分タグを週次レビュー画面にローカル表示する
  - **あわせて「AIコーチングにおける日記本文の扱い」(#12の残論点)を決着**: 設定画面に「日記の本文をAIに送る」トグル(`Settings.sendDiaryTextToAi`、デフォルトOFF)を新設し、ONの週だけ本文を`WeeklyDigest.diaryEntries`としてGeminiに送る。常時送信案は「本文はデフォルトで外部AIに送らない」という設計原則(AIコンサルティング設計書7章)を崩すため不採用、画面表示のみ(AI非連携)案はAIコメントを生活背景に沿わせられないため不採用とし、オプトイン方式を採用した。日記カードには本文がAIに渡る状態かどうかを常に注記する
- **2026-07-13: 週次レビューへのクロス分析追加(#112) → 「睡眠×食事」「気分×食事」「飲酒×食事」の3軸で実装。** 蓄積済みの複数系列(体重・食事・日記・Garmin活動)を掛け合わせ、記録コストを増やさずに既存データの価値を高める。実装時の決定事項:
  - **飲酒を分析軸に使うため、`DiaryRecord`に飲酒タグ(boolean・気分タグと同列の任意入力)を追加した。** 飲酒はカロリーとして食事記録に載っていても「飲んだ日」というフラグにはならず、日記本文の自由記述はAI送信オプトイン制のため集計に使いにくい。晩酌の有無は1日単位の事実で日記の1日1件モデルと一致するため、日記側に持たせた。トグルOFFは「飲酒なし」ではなく「記録なし」(undefined)として扱い、集計の「それ以外の日」もタグの無い日全体として表現する
  - **睡眠×食事のペアリングは「翌日」ではなく同じ日付にした。** Issue本文の候補は「睡眠時間×翌日の摂取カロリー」だったが、Garminの睡眠時間は「その日の朝までの夜間睡眠」を指すため、同日の食事がまさに「睡眠不足明けの日の食事」に相当する(日付+1でペアリングすると1日ズレる)
  - **見せ方は断定を避けた事実の提示のみとし、解釈はAIコメントに委ねる。** 週単位(最大7日)はサンプル数が少なく「相関」とは言い切れないため、画面は「睡眠6時間未満の日が2日あり、その日の平均摂取カロリーは…」形式+「差は偶然の範囲のこともある」注記に留め、AIプロンプトにも「相関・因果を断定しない」「飲酒を責めるトーンにしない」制約を追加した
  - 候補にあった「歩数・消費カロリー×体重変化」の軸は見送り(実測TDEEとGarmin計測消費の突き合わせ(#82)で既に同じ目的をカバーしているため)
  - シートの飲酒列は日記記録タブのID列(E)の後ろのF列に置く(既存シートのID列を動かさない。種目マスタの部位列(#104)と同じ方針)
- **2026-07-14: 月次レビューを追加した(#114) → 週次(#45)と役割を分け、俯瞰専用の内容に絞って実装。** 週次は「今週の達成→来週のアクション」で行動を変えるレビュー、月次はペースの加速・減速や目標までの残距離を見る俯瞰、と役割を分けた。実装時の決定事項:
  - **月の定義を「その月に日曜(週の終わり)が含まれる月曜始まりの週の集合」とした(4〜5週)。** 暦月は月曜始まりの週と端が揃わない(月初・月末で週が分断される)ため、週を分割せず日曜の属する月へ丸ごと割り当てる割り切りを採った。これにより全ての週がちょうど1つの月に属し、隣接月と重複・欠落しない。代償として月次の期間境界は暦月とズレる(例: 2026年7月は6/29〜7/26で、7/27〜31は8月レビューに入る)が、週平均体重・週次TDEE逆算という「週を最小単位とする既存の集計」をそのまま再利用できる利点を優先した。実装は`src/lib/date.ts`の`weekStartsOfMonth`
  - **UIはサブタブを増やさず、レビュータブ内に「週/月」の`SegmentedControl`を置いた(Issue本文の方針どおり)。** 推移画面のトップは「グラフ/履歴/レビュー」のまま、レビューの中で週次・月次を切り替える(タブの階層を深くしない)
  - **表示内容は月次ならではの項目に絞り、週次の全セクションは持ち込まなかった。** 採用: 週平均体重の推移(折れ線4〜5点)・目標へのマイルストーン(今月ペース維持時の到達見込み)・月間実測TDEE(週単位よりブレの少ない安定値)・カロリー/記録率・月窓クロス分析・月次AIコメント。見送り: PFC平均・活動・筋トレ・水分・日記(週次の粒度で足り、月次の俯瞰では冗長なため)
  - **月間の体重変化・ペースは「記録がある最初の週と最後の週の週平均」で計算した。** 端の週に体重記録が無くても、記録のある週同士の差を週数差で割ることでペースの意味を保つ(単純な暦月端の差にしない)。体重記録のある週が1週以下ならペース・見込みはnull(比較できないため)
  - **月次TDEEは月内の全有効週の週次逆算値の平均にした。** 週次レビューの「直近3週の平滑化」より窓が広く、目標カロリー補正提案(#44)の信頼度を上げる。週ごとの逆算値の幅(最小〜最大)も併記してブレを可視化する
  - **クロス集計・クロス分析カードは週次と共有した(`buildCrossAnalysis`を期間パラメータ化、`CrossAnalysisCard`を共通コンポーネント化)。** #112の集計をそのまま月幅で再実行する。月次は標本数が多いぶん注記を「傾向の目安」に変えるが、因果を断定しない原則は週次と同じ
  - **記録率低下フラグは「経過日数」を分母にした(月全体の総日数ではない)。** 進行中の当月は今日までに経過した日数、完了済みの月は総日数を分母に「7割未満」を判定する。月全体を分母にすると、当月の序盤で毎日記録していても常に「記録不足」警告が出てAI判定も歪むため。データ不足フラグ(記録7日未満)は絶対値のままにし、月序盤にまだ評価に足るデータが無いことは正しく伝える
  - **AIコメントは週次の仕組み(Gemini中継+キャッシュ)を月キーで流用した。** 出力契約(`WeeklyAdvice`)は共通で、winsを「今月の良かった変化」・actionsを「来月の重点」として使う。キャッシュは`MonthlyAdviceRecord`(月キー)。プロンプトだけ月次版(`worker/monthlyAdvice.ts`)にし、`weeks`系列からペースの加速・減速を読ませる。月次digestに日記本文は含めない(週次のオプトインは月次には設けていない)
- 2026-07-06: ホーム画面での表示順 → 体重・食事・筋トレ・水分・日記から、体重・食事・水分・日記・筋トレの順に変更した(ユーザーの希望により筋トレセクションを最後尾に)。画面設計書の章立てもこの順に並び替えた

## 技術スタック・インフラ

- 技術スタックの選定 → React + Vite + PWA、IndexedDB(ローカルファースト)、Cloudflare Workers中継の構成で決定し、実装済み(要件定義書6章)。当初検討したReact + Spring Bootのフルスタック構成、クラウド同期(Firebase/Supabase等)は以下の理由で見送り:
  - Render/RailwayなどのPaaS無料枠は、DBの自動削除やスリープ挙動があり、日次記録アプリの運用には不向き(2026年時点の調査結果より)
  - Oracle Cloud Always Freeによる自前サーバー運用はコストゼロだが、運用の手間・アカウント審査等の不確実性がある
  - 「PC閲覧はたまに確認できれば十分」という要件確認により、常時同期のためのサーバー・DBを持つ必要がなくなった
- ホスティング → 当初想定のCloudflare Pagesから、AI判定・同期の中継Workerと同一のWorker(`wrangler.toml` の `[assets]`)で静的配信までまとめる構成に変更
- オフライン/オンラインの判定方法 → `navigator.onLine` のみで判定する(要件定義書4.3章)
- **2026-07-11: Worker APIの保護(#87) → 共有トークン方式(Workerシークレット`API_AUTH_TOKEN` × 設定画面の`apiToken`のBearer照合)で決定・実装済み。** リポジトリ全体レビューで「無認証のAPIから健康データの読み出し・Gemini課金の発生が可能」と指摘されたため、マルチユーザー化(#67)を待たず単独で先行実装した。単一ユーザー前提の最小構成。シークレット未設定時は無認証で通す(移行期間・ローカル開発への配慮)
- **2026-07-11: マルチユーザー化の方針検討(#67) → 費用対効果が見合わないと判断し対応不要としてクローズ。** Issue内で行った事業性評価の結論: 食事・体重記録アプリは競合(あすけん等)が「AI写真判定+管理栄養士コメント+巨大な食品DB+PFC管理」を月額300円前後で既に提供しており、個人開発の構造的弱点(食品DB・AI精度)に直撃するうえ技術的な堀にならない。加えて共有ホスティング・SaaS化には認証基盤・課金・法務(特定商取引法・健康データは要配慮個人情報)対応が必要でコストに見合わない。当面は「本人+知り合いへのBYOK/無料配布」を前提とし、共有ホスティング・SaaS化は行わない。Worker API認証(#87)は本論点と切り離し、無認証API自体のセキュリティリスク対策として既に実装済みのまま維持する
- **2026-07-11: デプロイパイプラインへのテスト実行組み込み(#18) → 費用対効果が見合わないと判断し対応不要としてクローズ。** GitHub Actions移行(push→test→build→deployのワークフロー化)や簡易案(Cloudflare Build commandを`npm run test && npm run build`に変更)を検討したが、個人開発・単一ユーザー向けというプロジェクト規模には見合わないと判断。デプロイ前に開発者自身が`npm run test`/`npm run build`をローカルで確認する現状の運用を継続する

## データ同期(スプレッドシート)

- 同期先・フォーマット(#3) → アプリのデータモデルに沿った新規専用スプレッドシート(体重記録/食事記録の2タブ)で決定。既存の手動運用シートは「1日1行、その日の食事の合計カロリー/PFC」という集計形式でアプリの`MealRecord`(1食事=1レコード)と粒度が合わないため、合流を見送った。認証方式はGoogleサービスアカウント(Workers側のみが保持)、書き込みAPIはGoogle Sheets API v4
- 同期失敗時のリトライ間隔・エラー通知の要否(#3) → 新しい仕組みは追加せず、既存の「未同期フラグを維持し次回起動時 or 手動ボタンで再試行」+ 設定画面でのエラーメッセージ表示のみで十分と判断
- **2026-07-12: 自動同期のトリガー拡充(#105) → 起動時のみ→「起動時+アプリ復帰時(`visibilitychange`)+オンライン復帰時(`online`)」に拡充・実装済み。** スマホPWAは「閉じずにホーム画面へ戻り数日後に復帰する」使い方が多く、起動時1回では未同期が溜まりやすいため。連続発火対策として自動トリガーに最小間隔5分のスロットリングと実行中の再入抑止を設けた(`src/sync/autoSync.ts`)。オンライン復帰は「同期が可能になった」明確なシグナルのため間隔チェックを飛ばし、オフラインでスキップした試行は間隔に数えない。定期ポーリング(setInterval)は、イベント駆動で実用上足りるうえ電池・API呼び出しの無駄になるため採用しなかった
- 同期済み記録の編集・削除の反映(#30) → ID列をキーにした upsert(編集)+ 削除トゥームストーン + `deleteDimension`(削除)で決定・実装済み(画面設計書10章)
- **2026-07-11: 水分・筋トレ・日記記録をスプレッドシート同期の対象に含めるか(#72) → 3つとも含めることに決定・実装済み。** 新規タブを1つずつ追加し、体重・食事と同じID列upsert方式で反映する(画面設計書10章)。あわせて以下2点を決定:
  - 日記の自由記述本文もそのまま同期する(気分タグのみに絞らない)。プライバシー面は認識した上で、バックアップの完全性を優先した
  - 筋トレは「その日の全セットをまるごと置き換え」という既存の保存方式(#8)を変えず、置き換えのたびに全セットへ新しいIDを振り直す。置き換え前の全セットIDを削除トゥームストーンとして残すことで、シート側の古いセット行を消す。1回の保存が「その日の既存行を全削除→新規行を全追記」という比較的重い同期になるが、他の記録と同じupsert/削除の仕組みをそのまま流用できることを優先した
- **2026-07-11: ローカルデータ消失リスクに備えたバックアップ促し通知(#14) → 対応不要と判断しクローズ。** 起動時の自動同期(`App.tsx`)+設定画面の手動同期ボタン+シートからの取り込み・復元(#54)により、ユーザーが手動でバックアップ操作を行う前提が既に無くなっており、「エクスポートを促す通知」という当初の課題設定が成立しなくなったため。ただしシートに同期されない設定・食事マスタ・種目マスタ・食事のAI推定値/写真参照は引き続きバックアップ対象外であり、このうち食事マスタ・種目マスタはインポート・エクスポート機能(#96)で別途対応する
- **2026-07-12: 食事マスタ・種目マスタのインポート・エクスポート機能(#96) → CSV/JSONファイル方式ではなく、既存のスプレッドシート同期・取り込みへの統合で決定・実装済み。** 検討結果(詳細は https://github.com/tatsuki73/lifelog/issues/96#issuecomment-4949599412 ):
  - ファイル方式はユーザーが手動でエクスポートしない限りバックアップされないが、既存同期に乗せれば起動時の自動同期でマスタも常時バックアップされ、#14から引き継いだ残課題が根本的に解消する。一括登録もシートのマスタタブに行を直接書いて「シートから取り込み」すればよく、ファイルアップロードUIが不要になる
  - マスタ専用スプレッドシートの新設案は不採用(スプレッドシートIDの追加管理・Workerのマルチスプレッドシート対応が必要になる割に、同一シート内のタブ分離で足りる以上のメリットがないため)。既存の同期用シートに「食事マスタ」「種目マスタ」の2タブを追加し、タブが無ければ同期時にWorkerが見出し行付きで自動作成する
  - 取り込みは記録と同じ「追加のみ・ローカル優先」に加え、IDが違っても同名(前後空白無視)の既存品目・種目をスキップする(種目マスタは名前がサジェストのキーで同名を許さないため)
  - コード埋め込みの定番メニューシードデータ(`foodMasterSeedData.ts`)と「定番メニューを一括登録」ボタンは撤去し、データはシート貼り付け用サンプル `docs/食事マスタ_定番メニューサンプル.csv` へ退避した

## Garmin連携・外部データ

- **2026-07-11: 既存ツールからのデータ連携方針(#11) → Garmin連携を実装して決着(#80〜#82)。** 検討結果(詳細は https://github.com/tatsuki73/lifelog/issues/11#issuecomment-4946239901 ):
  - ライブラリはPythonの `python-garminconnect` を採用(2025年頃のGarmin認証基盤刷新に追随済みで活発にメンテされている。Node系の `garmin-connect` は更新停止のため不採用。公式のGarmin Health APIは法人向け審査が必要で個人利用には非現実的)
  - アーキテクチャはGitHub Actionsのcron(毎日1回)で前日分を取得し、同期用スプレッドシートの新タブ「活動記録」(1日1行)へ日付キーでupsert。アプリ側は既存の「シートから取り込み」(`runImport`)を拡張してこのタブも読む(取り込み専用・常にシート側で上書き。#81)
  - UIは週次レビュー(`WeeklyDigest.activity`+逆算TDEEとの突き合わせ)・推移グラフ(歩数・睡眠)・履歴(読み取り専用の「活動」種別)に統合(#82)。ホームは「今日」のデータが出せないため対象外
  - 非公式APIのため壊れうる前提とし、壊れても日付範囲指定のバックフィルで復旧できる設計にした
  - あすけん側は実用的な非公式APIが無く、手動継続 or 本アプリへの移行が現実的だったが、Garmin連携により活動量・消費カロリー側のデータ一元化は達成できたため、あすけんからのデータ移行は対応不要と判断した(#11はこの結論を含めてクローズ済み)

## フェーズ4機能(生活ログへの拡張)

- **2026-07-14: フェーズ4(生活ログへの拡張)を新設し、血圧記録(#117)・周囲径記録(#118)・汎用の習慣トラッカー(#113)を実装した(2026-07-13の検討より)。** いずれも「からだログ」のスコープ(体の記録)を超え始める拡張で、既存パターン(データ層1ファイル+単一ファイルの記録画面+シート1タブの同期・取り込み)に小さく乗せる方針とし、アプリの土台は作り替えない。要件定義書3章のフェーズ表・4.8章に追記した。
- **血圧記録(#117)のデータモデル** → 体重記録と同型(1日1件・日付キー・後勝ち)。最高血圧・最低血圧を必須、脈拍を任意とした。家庭血圧の基準に合わせ「朝の測定値」を記録する運用を想定(体重測定と同じ朝のリズムに揃う)。
- **血圧の医療上の線引き(#117。重要)** → アプリは医療機器ではないため、警告は「家庭血圧135/85以上の日が◯日」のような事実の提示に留め、コード・画面・AIのいずれも医学的判断(診断・受診勧奨・薬の提案)をしない方針で決定。閾値135/85は高血圧治療ガイドラインの家庭血圧の値を「高値の目安」として使うだけで、判定・断定には使わない。血圧由来の新しい警告フラグ(`DigestFlag`)は設けない(高値の判断は医学的評価にあたるため)。AIコーチングにも同じ免責をプロンプトで課した(AIコンサルティング設計書5章)。
- **「体重×血圧」の見せ方(#117。#112のクロス分析の軸)** → 既存のクロス分析(`crossAnalysis`)は「各条件 × 摂取カロリー」の構造で血圧と噛み合わないため、`crossAnalysis` を作り替えるのではなく、`WeeklyDigest.bloodPressure` に同じ週の週平均体重(`weekAvgWeightKg`)を併記して体重と血圧を並べて見せる形にした。医学的に確立した体重×血圧の関係を「事実の並置」で提示する。月次では週平均体重の併記はせず、体重の推移は月次の体重セクションで見せる。
- **周囲径記録(#118)のスコープ** → まずは腹囲を必須とする最小構成に、胸囲・太ももを任意項目として加えた(将来の拡張余地を型に残す)。入力頻度が月1程度と体重・血圧と異なるため専用の記録画面を持ち、推移グラフは点が疎でも破綻しないよう1点のときは線を引かず点のみ描く。低頻度のため推移グラフは期間フィルタに関わらず常に全期間を表示する。週次レビューへの統合は入力頻度が合わないため見送り、月次レビュー(#114)への統合は今後検討とした。
- **習慣記録(#113)の主キー設計** → 「1日1習慣1件・後勝ち」を`put()`だけで成立させるため、`id = ${date}_${habitId}` の合成キーにした(体重・日記の日付キーと同じ考え方を2軸に拡張)。チェックを外す操作は記録の削除(トゥームストーン)で表し、記録の存在が「その日にやった」を意味する設計にした(`done`のような真偽フラグは持たない)。習慣名はシートの可読性のために`HabitRecord`へ非正規化コピーする(筋トレの`exerciseName`と同じ)。薬・サプリの服用記録(#113コメント)は専用機能を作らず、汎用の習慣として扱う(「血圧の薬」「プロテイン」等を習慣マスタに登録)。
- **フェーズ4タブの自動作成(#117・#118・#113)** → 血圧・周囲径・習慣マスタ・習慣記録の4タブは後付けで既存スプレッドシートに存在しないため、マスタ2タブ(#96)と同じく初回同期時にWorkerがヘッダー行付きで自動作成する方式にした。記録5タブ(体重・食事・水分・筋トレ・日記)の「手動作成が前提」という扱いは変えていない(これらはフェーズ1〜2から存在するため)。旧Workerとの後方互換のため、`SyncPushResult`/`SyncPullResult` に追加した同期済みID・取り込みレコードのフィールドはすべて省略可(`?`)とし、未対応の旧Workerが返さなくてもクライアントは空として扱う(マスタ2タブ導入時と同じ方針)。
