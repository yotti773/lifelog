# からだログ 意思決定ログ

作成日: 2026-07-06
関連ドキュメント: からだログ_要件定義書.md、からだログ_画面設計書.md

要件定義書・画面設計書に混在していた「決定済み論点(経緯の記録)」を本ログに集約する。**本ログは経緯の記録であり、現在の仕様の真実の情報源は要件定義書・画面設計書の本文**である(本文と食い違う場合は本文が正)。新しい決定は末尾ではなく該当テーマの節に追記し、日付が分かるものは日付を添える。

## 全体方針・フェーズ

- **2026-07-06: アプリの最終形を「必要なことを入力するとパーソナルなコンサルティングができるアプリ」と再定義した(フェーズ3)。** 記録(現状把握)だけでなく「記録 → 評価 → 助言 → 行動修正」のループを回すことを長期目標とする。フェーズ3の機能構成は以下で決定(要件定義書4.7章):
  - 採用: 身体プロフィール+目標カロリー自動計算(#43)、実測TDEE推定(#44)、週次レビュー(#45)、記録率の可視化(#46)、PFC目標値(#47)、AIコーチング(#12)
  - 見送り: 活動データ(歩数・運動消費)の手入力欄の先行実装 → 実測TDEEの逆算(#44)があれば活動量データが無くても収支評価は成立するため、Garmin/Google Fit連携(#11・#13)に統合し単独では実装しない
- **2026-07-06: AIコーチングの実現方式** → 数値の計算・集計・安全判定はすべてコード側で決定論的に行い(`WeeklyDigest`)、AIには解釈と言語化だけをさせる構成で決定。モデルは低コストな軽量モデル(Gemini Flash-Lite級)を既存のWorker中継で呼び出す。詳細はAIコンサルティング設計書を参照
- 2026-07-06: ホーム画面での表示順 → 体重・食事・筋トレ・水分・日記から、体重・食事・水分・日記・筋トレの順に変更した(ユーザーの希望により筋トレセクションを最後尾に)。画面設計書の章立てもこの順に並び替えた

## 技術スタック・インフラ

- 技術スタックの選定 → React + Vite + PWA、IndexedDB(ローカルファースト)、Cloudflare Workers中継の構成で決定し、実装済み(要件定義書6章)。当初検討したReact + Spring Bootのフルスタック構成、クラウド同期(Firebase/Supabase等)は以下の理由で見送り:
  - Render/RailwayなどのPaaS無料枠は、DBの自動削除やスリープ挙動があり、日次記録アプリの運用には不向き(2026年時点の調査結果より)
  - Oracle Cloud Always Freeによる自前サーバー運用はコストゼロだが、運用の手間・アカウント審査等の不確実性がある
  - 「PC閲覧はたまに確認できれば十分」という要件確認により、常時同期のためのサーバー・DBを持つ必要がなくなった
- ホスティング → 当初想定のCloudflare Pagesから、AI判定・同期の中継Workerと同一のWorker(`wrangler.toml` の `[assets]`)で静的配信までまとめる構成に変更
- オフライン/オンラインの判定方法 → `navigator.onLine` のみで判定する(要件定義書4.3章)
- **2026-07-11: Worker APIの保護(#87) → 共有トークン方式(Workerシークレット`API_AUTH_TOKEN` × 設定画面の`apiToken`のBearer照合)で決定・実装済み。** リポジトリ全体レビューで「無認証のAPIから健康データの読み出し・Gemini課金の発生が可能」と指摘されたため、マルチユーザー化(#67)を待たず単独で先行実装した。単一ユーザー前提の最小構成であり、マルチユーザー化の際は本格的な認証(OAuth等)に置き換える。シークレット未設定時は無認証で通す(移行期間・ローカル開発への配慮)

## データ同期(スプレッドシート)

- 同期先・フォーマット(#3) → アプリのデータモデルに沿った新規専用スプレッドシート(体重記録/食事記録の2タブ)で決定。既存の手動運用シートは「1日1行、その日の食事の合計カロリー/PFC」という集計形式でアプリの`MealRecord`(1食事=1レコード)と粒度が合わないため、合流を見送った。認証方式はGoogleサービスアカウント(Workers側のみが保持)、書き込みAPIはGoogle Sheets API v4
- 同期失敗時のリトライ間隔・エラー通知の要否(#3) → 新しい仕組みは追加せず、既存の「未同期フラグを維持し次回起動時 or 手動ボタンで再試行」+ 設定画面でのエラーメッセージ表示のみで十分と判断
- 同期済み記録の編集・削除の反映(#30) → ID列をキーにした upsert(編集)+ 削除トゥームストーン + `deleteDimension`(削除)で決定・実装済み(画面設計書10章)
- **2026-07-11: 水分・筋トレ・日記記録をスプレッドシート同期の対象に含めるか(#72) → 3つとも含めることに決定・実装済み。** 新規タブを1つずつ追加し、体重・食事と同じID列upsert方式で反映する(画面設計書10章)。あわせて以下2点を決定:
  - 日記の自由記述本文もそのまま同期する(気分タグのみに絞らない)。プライバシー面は認識した上で、バックアップの完全性を優先した
  - 筋トレは「その日の全セットをまるごと置き換え」という既存の保存方式(#8)を変えず、置き換えのたびに全セットへ新しいIDを振り直す。置き換え前の全セットIDを削除トゥームストーンとして残すことで、シート側の古いセット行を消す。1回の保存が「その日の既存行を全削除→新規行を全追記」という比較的重い同期になるが、他の記録と同じupsert/削除の仕組みをそのまま流用できることを優先した

## Garmin連携・外部データ

- **2026-07-11: 既存ツールからのデータ連携方針(#11) → Garmin連携を実装して決着(#80〜#82)。** 検討結果(詳細は https://github.com/tatsuki73/lifelog/issues/11#issuecomment-4946239901 ):
  - ライブラリはPythonの `python-garminconnect` を採用(2025年頃のGarmin認証基盤刷新に追随済みで活発にメンテされている。Node系の `garmin-connect` は更新停止のため不採用。公式のGarmin Health APIは法人向け審査が必要で個人利用には非現実的)
  - アーキテクチャはGitHub Actionsのcron(毎日1回)で前日分を取得し、同期用スプレッドシートの新タブ「活動記録」(1日1行)へ日付キーでupsert。アプリ側は既存の「シートから取り込み」(`runImport`)を拡張してこのタブも読む(取り込み専用・常にシート側で上書き。#81)
  - UIは週次レビュー(`WeeklyDigest.activity`+逆算TDEEとの突き合わせ)・推移グラフ(歩数・睡眠)・履歴(読み取り専用の「活動」種別)に統合(#82)。ホームは「今日」のデータが出せないため対象外
  - 非公式APIのため壊れうる前提とし、壊れても日付範囲指定のバックフィルで復旧できる設計にした
  - あすけん側は実用的な非公式APIが無く、手動継続 or 本アプリへの移行が現実的だったが、Garmin連携により活動量・消費カロリー側のデータ一元化は達成できたため、あすけんからのデータ移行は対応不要と判断した(#11はこの結論を含めてクローズ済み)
- **2026-07-11: Google Fit連携(#13) → 対応不要と判断しクローズ。** 歩数・消費カロリーの自動取得というニーズはGarmin連携(#80〜#82)で既に満たされており、同種のデータソースを二重に連携する必要性がないため

## 体重記録・推移画面

- **2026-07-11: 水分・筋トレ・日記の履歴確認(#73) → 既存の履歴画面(推移>履歴)の種別サブタブを6択(体重・食事・水分・筋トレ・日記・活動)に拡張する方式で決定・実装済み。** 水分・筋トレは日付ごとに集約した1日1行、日記は1日1件をそのまま1行で表示し、タップで各記録画面の該当日に遷移する(日記は`?date=`パラメータで過去日を編集可能にした)

- 体重の増減表示(前回比) → 体重入力欄の直下に表示する(画面設計書3章)
- 履歴確認画面の導線 → 推移画面内のサブタブ。期間フィルタは推移グラフと連動させず、全期間・日付降順の固定表示(画面設計書8.1章)
- 基準日ちょうどの体重記録が存在しない場合の進捗バー起点 → 一番古い体重記録にフォールバックする(画面設計書9章)

## フェーズ2機能(筋トレ・水分・日記)

- 筋トレ記録の粒度・種目管理・画面統合方針(#8) → セット単位(重量×回数をセットごとに記録)、種目マスタ+自由入力(食事マスタに準ずる仕組みだが重量・回数は持たない)、ホーム画面に体重・食事と並べて表示、推移グラフはMVPでは持たず一覧確認のみ、で決定(画面設計書7章・7.1章)
- 水分記録の記録単位・目標設定・画面統合方針(#9) → ml単位でクイック追加ボタン+手入力、目標摂取量の設定+ホームでの差分表示あり、ホーム画面に体重・食事と並べて表示、推移グラフあり(カロリー推移と同様)、記録の訂正は削除→再記録のみ(編集UIは持たない)で決定(画面設計書5章)
- 日記の形式・記録粒度・画面統合方針(#10) → 自由記述+気分タグ(3段階)のハイブリッド形式、体重と同様に日付キーで1日1件(後勝ち)、ホーム画面に体重・食事・水分と並べて表示、写真添付はMVPでは行わない、で決定(画面設計書6章)
- **2026-07-07: フェーズ2の画面構成をClaude Design製プロトタイプ(docs/handoff_lifelog_app/prototype.dc.html)に合わせて変更した(#8〜#10の実装時)。** #41で決めたテキスト仕様をもとにデザインプロトタイプを作成・確認した結果、以下の3点を見た目・操作感を優先してデザイン側の構成で確定した:
  - 水分(#9): 「ホーム内でワンタップ完結+手入力モーダル」→「専用の記録画面(クイック追加 100/200/350/500mlの4ボタン+今日の記録リスト)。手入力は設けない」。ホームの水分行は合計+進捗バーの表示とタップ遷移のみ
  - 日記(#10): 気分タグを3段階(良い/普通/悪い)→5段階(絶好調/良い/普通/眠い/不調)に変更。編集画面の明示的な削除ボタンは持たず、本文・気分とも空で保存した場合に記録を削除する挙動とした
  - 筋トレ(#8): 「1画面1種目+セット逐次確定(＋ボタン)+前回値prefill+メモ+マスタ登録チェック」→「1画面で当日の複数種目・全セットをまとめて編集(種目カードの追加/削除、セット行の追加/削除)。保存は当日分の置き換え」。前回値prefill・メモ・マスタ自動登録は見送り、要否は画面設計書12章の論点として残す。種目マスタは種目名入力のサジェスト+設定画面からの管理画面として維持

## UI全般・バックアップ

- ホーム画面の記録導線 → フローティングアクションボタン+アクションシートの構成を廃止し、体重/体脂肪率カードおよび食事区分ラベルへの直接タップで対応する記録画面に遷移する方式に変更した(MUI移行 Issue #27)。カード・ラベルへの直接タップという明確な導線ができたことで、両者を選ばせるアクションシートは冗長と判断した
- JSONバックアップ(エクスポート/インポート)UI(#27) → MUI移行時に設定画面から削除した。ロジック(`db/backup.ts`)は残置し、必要になれば復元可能。手動移行の主目的はスプレッドシート同期でおおむね代替できるため(スプレッドシート側からの復元はできない点は把握した上での判断)。これに伴い、定期エクスポートのリマインド通知(#14)も保留とした
- スプレッドシートからの復元(#54) → #14(バックアップ促し通知)の検討過程で、「復元不可」だったスプレッドシート同期に手動取り込み(インポート)を追加し、シートを復元可能なバックアップ兼過去データ移行の入口に位置づけ直した。JSONバックアップUIは引き続き再追加しない(シートに無いAI推定値・食事マスタ・設定まで含めた完全バックアップが必要になった場合のみ`db/backup.ts`の再利用を検討)。取り込みのマージは「追加のみ・ローカル優先」とし、シート側の編集をローカルに反映する双方向同期にはしない(画面設計書10.1章)
