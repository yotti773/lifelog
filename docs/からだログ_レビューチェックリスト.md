# からだログ リポジトリレビュー・ドキュメント整合チェックリスト

作成日: 2026-07-11(Issue #86 のリポジトリ全体レビューで使った観点を手順化)
関連ドキュメント: CLAUDE.md、からだログ_要件定義書.md、からだログ_画面設計書.md、からだログ_意思決定ログ.md

## 目的・使い方

リポジトリ全体の健全性確認と「仕様書↔実装のズレ」の検出を、**判断の少ない機械的な手順**として定義する。実装が進むたびにドキュメントは古くなるので、大きめの機能追加が続いた後や、依頼を受けたタイミングでこのチェックリストを上から順に実行する。

- 各項目は「実行するコマンド/見る場所」と「期待する状態」のセット。期待と違ったら**修正せずにまず記録**し、最後にまとめてIssue化 or その場の指示に従って修正する
- 修正する場合もこのリポジトリの開発フロー(CLAUDE.md)に従う: 非自明な作業はIssueを立ててから、ブランチ上で作業し、テスト・ビルドで検証する

## 1. 事前準備

1. `CLAUDE.md` と `docs/` の仕様書5点(要件定義書・画面設計書・デザインガイド・AIコンサルティング設計書・意思決定ログ)を読む
2. `npm install` を実行する(クローン直後は `vitest: not found` になるため必須)

## 2. 健全性チェック

| 手順 | 期待する状態 |
|---|---|
| `npm run test` | 全テストがパスする(1件でも失敗したら最優先で記録) |
| `npm run build` | 成功する。「Some chunks are larger than 500 kB」の警告は既知(バンドル約800kB)で問題ではない |
| `npx tsc -p worker/tsconfig.json` | エラーなし(Workerはsrc/と独立ビルドのため別途チェックが必要) |
| `npm run lint` | **動かないのが正常**(ESLint未導入。CLAUDE.md参照)。動かないことを「壊れている」と報告しない |

## 3. ドキュメント↔実装の整合チェック

「仕様書に書いてある状態」と「コード・Issueの現状」を突き合わせる。過去に実際にズレていた箇所ばかりなので、毎回全部確認する。

### 3.1 フェーズ・実装状況の表記

- 要件定義書3章のフェーズ表の「状態」列(実装済み/未実装)が、`src/pages/` に画面が存在するかと一致しているか
- 画面設計書の各章にある「※実装状況:」の注記・「今後検討」「未決定」の記述について、対応するIssueがクローズ済みでないかをGitHubで確認する(クローズ済みなら本文を現在の仕様に書き換え、経緯を意思決定ログへ)

### 3.2 データモデル

- 画面設計書11章のデータモデルと `src/types.ts` のinterfaceを1つずつ突き合わせる(エンティティの欠落・フィールドの差分)
- `src/db/db.ts` のテーブル一覧と11章のエンティティ一覧が一致するか

### 3.3 同期まわり(ズレやすい)

- `worker/sheetsSync.ts` の `*_CONFIG`(タブ名・ID列)と、画面設計書10章の列構成・CLAUDE.mdのID列の記述が一致するか
- `worker/sheetsImport.ts` が読むタブ数(`handleImportSheets` 内の `readSheetRows` 呼び出し+活動記録)と、画面設計書10.1章・CLAUDE.mdの「◯タブ」の記述が一致するか
- `worker/` 内のローカル型(`*Input`)と `src/types.ts` / `src/sync/types.ts` の型がフィールドレベルで一致するか(worker/は手動同期。sheetsSync.ts冒頭のコメント参照)

### 3.4 技術スタック・構成の記述

- README・CLAUDE.mdが参照するファイルが実在するか(過去の例: MUI移行後も `tailwind.config.js` 前提の記述が残っていた)。`package.json` のdependenciesと技術スタックの記述(MUI/Tailwind等)が一致するか
- READMEのWorkerエンドポイント一覧と `worker/index.ts` のルート(`url.pathname === "/api/..."` をgrep)が一致するか
- READMEのディレクトリ構成・開発コマンドが `package.json` の `scripts` と一致するか

### 3.5 オープンな論点とIssueの対応

- 要件定義書7章・画面設計書12章に挙がっている論点のIssue番号を `gh issue view <番号>`(またはGitHub MCP)で確認し、クローズ済みの論点が「オープン」として残っていないか
- 逆に、Open Issueのうち仕様に影響するもの(仕様変更・新機能)が仕様書に反映されているか
- Issueに `優先度: 高/中/低` ラベルが付いているか(付いていないものは記録)

## 4. コード不変条件チェック(grepでできる)

CLAUDE.mdに書かれた「やってはいけないこと」の機械的な検査。ヒットしたら該当箇所を読んで、正当な例外(下記)か違反かを判定する。

| チェック | コマンド | 合格条件 |
|---|---|---|
| `synced: true` の直接セット | `grep -rn "synced: true" src/` | ヒットは `markXxxSynced` 系関数・`runImport`(シート由来の保存)・テストのみ |
| 絵文字の混入 | `grep -rnP "[\x{1F300}-\x{1FAFF}\x{2600}-\x{27BF}]" src/ index.html` | ヒット0件(アイコンは `src/components/icons.tsx` のSVGのみ) |
| accent色(#FFC145)の誤用 | `grep -rn "FFC145\|accent" src/` | 使用箇所が「達成の瞬間の演出」か「PFCのF系列色」(theme.tsの注記)のみ |
| `_redirects` の追加 | `ls public/_redirects` | 存在しない(あるとCloudflareのデプロイが無限リダイレクトで失敗する) |
| WorkerでのローカルTZ依存 | `grep -n "getHours\|getMonth\|getDate" worker/` | ヒット0件(WorkerはUTC動作。JST変換は `Intl.DateTimeFormat` + `timeZone: "Asia/Tokyo"` のみ) |
| クライアントへのシークレット漏れ | `grep -rn "VITE_" src/ vite.config.ts` と `grep -rn "PRIVATE_KEY\|API_KEY" src/` | APIキー・秘密鍵・スプレッドシートIDがクライアントコードに現れない |
| Dexieスキーマの破壊的変更 | `git diff main -- src/db/db.ts`(差分レビュー時) | 既存の `db.version(n)` を書き換えず、新しいversionの追加のみ |
| booleanのインデックス化 | `src/db/db.ts` のstores定義 | `synced` がインデックスに入っていない(IndexedDBはbooleanをキーにできない。JS側filterが正) |

## 5. UIコードのレビュー観点(新規・変更画面を見るとき)

- **`useLiveQuery` の undefined/null**: `.first()`/`.last()`/`.get()` など「見つからない」に解決しうるクエリは、結果を `?? null` で正規化してから返しているか(`undefined` のままだとローディング分岐から抜けられない。CLAUDE.mdの注意点参照)
- 数値表示の丸め: PFCなど小数を合算する値は表示前に丸めているか(過去の例: Issue #52の浮動小数点誤差)
- 固定幅レイアウトに桁数が増える値を入れていないか(過去の例: Issue #78のPFC 3桁折り返し)。数値が3〜4桁になるケースを想定して見る
- UI変更はスクリーンショットで確認する: `npm run dev` を起動し、Playwright(スクラッチディレクトリに `npm install playwright`、chromiumは `/opt/pw-browsers/chromium` 等の既存インストールを利用)で該当画面を撮る。データが必要な画面は `page.evaluate` で `indexedDB.open("lifelog")` を開きレコードを直接putすると再現できる

## 6. セキュリティ観点

- `worker/index.ts` の全 `/api/*` ルートが共有トークン認証(`worker/auth.ts`、Issue #87)を通っているか。**新しいエンドポイントが認証チェックの後ろに追加されているか**を必ず確認
- 新しいAPI呼び出しをクライアントに足すときは `src/api/apiAuth.ts` のヘッダを付けているか
- シークレット類(Gemini APIキー・サービスアカウント・`API_AUTH_TOKEN`)がWorkerシークレット/GitHub Secrets以外の場所(コード・docs・コミット)に書かれていないか

## 7. 結果のまとめ方

1. 発見事項を「重要度(高=データ破壊・セキュリティ・全損系 / 中=機能不全・ドキュメントの実害あるズレ / 低=表記ゆれ・軽微)」で分類する
2. ドキュメントのズレはまとめて1つのIssue(documentationラベル)に、コードの問題は個別のIssueにする。優先度ラベルの基準はCLAUDE.mdの開発フロー参照
3. 修正まで行う場合はIssueごとにコミットを分け、コミットメッセージでIssue番号を参照する
