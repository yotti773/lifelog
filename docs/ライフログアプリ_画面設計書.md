## 1. 全体構成(ナビゲーション)

ボトムナビゲーション3タブ構成とする。

| タブ | 役割 |
|---|---|
| ホーム | 起動時に表示。その日の体重・食事記録を時系列でまとめて表示 |
| 推移 | 体重・カロリーのグラフ、目標までの進捗確認 |
| 設定 | 目標体重/目標日の設定、JSONエクスポート/インポート |

記録の入力は、ホーム画面右下の「＋」ボタン(フローティングアクションボタン)から「体重を記録」「食事を記録」をアクションシートで選択して行う。

## 2. ホーム画面

### 表示内容
- 当日の日付
- 体重・体脂肪率(当日1件のみ表示。同日に複数回記録した場合は最後に保存した値で上書きされる=後勝ち。体脂肪率は未記録なら非表示)
- 食事記録一覧(朝食/昼食/夕食/間食の4区分。未記録の区分は「未記録」と表示)
  - 同じ区分に複数品目がある場合は区分行に合計カロリーを表示し、品目ごとの内訳(時刻・品目名・カロリー)をその下にサブ行で表示する
- 当日の摂取カロリー合計、目標カロリーとの差分(「あと◯◯kcal食べられます」、目標超過時は「◯◯kcalオーバーしています」の形で表示)
- 当日の摂取PFC合計(たんぱく質/脂質/炭水化物、それぞれグラム表示)
- 各記録の同期状態(スプレッドシートへ未同期の場合は「未同期」のラベルを控えめに表示。7章の同期フロー参照)※未実装。同期機能本体(Issue #3)と合わせて対応する。現時点の未同期状況は設定画面の未同期件数表示で確認する

## 3. 体重記録フロー

1. ホーム画面の「＋」→「体重を記録」を選択
2. 体重入力画面が開く
   - 日時: 現在時刻がデフォルト(タップで変更可能)
   - 体重: 数値入力欄で入力(スマホでは数値キーボードが開く)。入力すると前回記録との差分(前回比 ±◯.◯kg)を入力欄の下に小さく表示する
   - 体脂肪率(任意): 体重と同様の数値入力欄。体組成計での計測値がない日は未入力のままでよい
   - メモ(任意): 「筋トレ後」「飲み会翌日」など一言メモ欄
3. 保存ボタンで確定 → ホーム画面に戻り、当日の一覧に反映される

- 1日1件のみ記録する仕様とする。同日に2回目以降を保存した場合は、その日の記録を新しい値で上書きする(後勝ち)

## 4. 食事記録フロー

記録は1画面で完結する(区分選択・AI判定結果・手動修正を別画面に分けない。9章にあった「1画面にまとめるか2画面に分けるか」の論点は1画面統合で決定済み)。

1. ホーム画面の「＋」→「食事を記録」を選択
2. 食事記録画面(1画面)で以下を入力する:
   - 区分: 朝食/昼食/夕食/間食(現在時刻から一番近い区分をデフォルトで選択状態にしておく)
   - 日時: 現在時刻がデフォルト(変更可能)
   - 入力フォーム: 料理名・カロリー・PFC(たんぱく質/脂質/炭水化物)を直接入力・修正する欄。写真判定・マスタ選択の結果もこのフォームに反映されるため、どの入力方式でも保存前にここで必ず確認・修正できる
3. 手入力の代わりに、同じ画面内の以下のセクションからフォームへ値を入れることもできる(画面上の並び順は 手入力フォーム → 写真 → マスタ):
   - **写真から記録する**: 補足情報(任意のテキスト。例: 「唐揚げ弁当、ご飯少なめ」)を添えて、撮影またはライブラリから写真を選択 → AI判定結果(料理名・カロリー・PFC)がフォームにprefillされる。複数の料理が写っている等で判定の自信が低い場合は、その旨の注意文言を表示する
   - **よく食べるものから選ぶ**: 食事マスタを品目名で検索でき、一覧はページネーション(6件/ページ)で表示する。選ぶと登録済みの固定値がフォームにprefillされる
4. 複数品目をまとめて記録する場合は「この品目をリストに追加してもう1品記録する」でリストに溜めてから、最後にまとめて保存する(同じ区分・同じ日時の複数レコードとして保存され、区分ごとの合計カロリー・PFCはホーム画面で集計される)
5. 「この内容をマスタに登録する」チェックボックス(オンで保存時に食事マスタへ追加登録される)
6. 保存ボタンで確定 → ホーム画面に戻り、当日の一覧に反映される

- 同じ区分(例: 昼食)に複数品目を記録したい場合は、上記4のまとめて保存のほか、同じ区分で複数回記録してもよい
- 写真はAI判定への送信時に長辺1024pxへリサイズ・JPEG圧縮され、判定にのみ使われる。**現時点では写真自体を端末に保存しない**(保存するかどうかは9章の論点)
- 保存済みの食事記録の編集・削除UIは未実装(Issue #24)

### 4.1 食事マスタ管理

- 食事マスタ(登録済みの「よく食べるもの」)の一覧・編集・削除は設定画面から行う(6章参照)
- 一覧の各項目は名前・カロリー・PFCを表示し、その場でインライン編集または削除ができる
- 設定画面には、よく利用する外食チェーン・コンビニの定番メニュー(各社公式の栄養成分表を出典とするシードデータ)を一括登録するボタンを設ける(登録済みの品目はスキップされ、重複追加されない)
- 重複登録(似た名前・同じ商品が複数件になる)の自動マージは行わず、気づいたユーザーが手動で整理する想定とする

## 5. 推移画面

- 体重推移グラフ
  - 1日1値(後勝ちで確定した値)を折れ線表示
  - 切り替え: 期間(週/月/全期間)
- 体脂肪率推移グラフ
  - 体重推移グラフと同様、1日1値を折れ線表示。体重グラフの直下に並べて表示する
  - 体脂肪率は任意入力のため、未記録の日は欠損(グラフ上は線を繋げない)として扱う
- カロリー推移グラフ(日別の摂取カロリー合計、目標ラインを重ねて表示)
- PFC推移の表示要否・表示形式(積み上げ棒グラフ等)は今後検討(9章参照)
- 目標までの進捗表示
  - 画面上部に進捗バーを表示する(起点体重 → 目標体重を100%として、現在体重の到達度をバーで可視化。起点の決め方は6章参照)。バーには現在値と目標日までの残り日数を併記し、目標達成時はバーをaccent色にする(達成の瞬間の演出。デザインガイド参照)
  - 体重記録が1件もない場合は進捗バーの代わりに記録を促すメッセージを表示する
  - 現在のペースで10月末に到達しそうな体重の予測(簡易線形予測)※未実装(Issue #25)
- グラフ表示と履歴確認画面(5.1参照)は、画面内の「グラフ/履歴」サブタブ(セグメントコントロール)で切り替える

### 5.1 履歴確認画面

- 推移画面のグラフ表示と対になる、記録の一覧確認用画面。推移画面内の「グラフ/履歴」サブタブとして実装する(ナビゲーションのタブ数自体は増やさない)
- 期間フィルタは持たず、推移グラフの期間切り替えとは独立して、全期間を日付降順(新しい記録が上)でリスト表示する
- 1行あたりの表示項目: 日付、体重、体脂肪率(未記録の日は「-」表示)、メモ(あれば日付の下に表示)。前日比の表示は任意(現時点では未実装)
- 各行をタップすると、その日の体重記録の編集画面に遷移し、体重・体脂肪率・メモを修正できる(3章の体重記録フローの編集版)。日時は記録のキーになっているため編集不可とする

## 6. 設定画面

- 目標体重・目標日付の設定(初期値: 64kg / 2026-10-31)
- 基準日の設定(進捗バー用)
  - 進捗バーの起点となる日付を1つ設定する(初期値: 未設定。ダイエット開始日など任意の日付をユーザーが選択)
  - 基準日当日の体重記録が存在すればその値を起点(0%地点)とする。基準日が未設定の場合や、基準日当日の記録が存在しない場合は、**一番古い体重記録を起点にフォールバックする**(9章にあった論点は決定済み)
  - 進捗バーは 推移画面 に表示する(基準日体重 → 目標体重を100%として、現在体重の到達度をバーで可視化。5章参照)
  - 体重記録が1件もない場合は進捗バーを表示せず、記録を促すメッセージを表示する
- 1日の目標摂取カロリーの設定(手動入力。基礎代謝等からの自動計算は9章の論点)
- 食事マスタの一覧・編集・削除、シードデータの一括登録ボタン(4.1章参照)
- JSONエクスポート(全データをファイル出力、PCでの確認・バックアップ用)
- JSONインポート(エクスポートしたファイルを読み込み、機種変更時などに復元)
- 定期エクスポートのリマインド通知(任意、忘れ防止)※未実装(Issue #14)
- データ同期(スプレッドシート書き出し)
  - 「今すぐ同期」ボタン(手動同期のトリガー。押下時は同期中であることが分かるようローディング表示)
  - 最終同期日時の表示
  - 未同期件数の表示(任意、あれば同期の必要性が分かりやすい)
  - 同期失敗時はその旨をエラーメッセージとして表示し、次回同期のタイミングで自動的に再試行する

## 7. データ同期フロー

同期は「ローカルファースト」を維持したまま、ネットワーク接続時に未同期の記録をスプレッドシートへ書き出すための一方向(アプリ→スプレッドシート)の仕組みとする。

※実装状況: 実装済み(Issue #3)。クライアント側の同期エンジン(未同期レコードの抽出、部分成功のハンドリング、失敗時の未同期維持、起動時/手動のトリガー)に加え、送信先となるWorker→Google Sheetsのトランスポート(`workerSheetsTransport` → `POST /api/sync-sheets`)も実装済み。

1. トリガー
   - アプリ起動時: オンライン(`navigator.onLine` で判定)であれば、起動直後にバックグラウンドで未同期レコードの同期を自動的に試みる(ユーザー操作は不要)
   - 手動: 設定画面の「今すぐ同期」ボタンから、いつでも明示的に同期を実行できる
2. 対象データ
   - `synced: false` の体重記録・食事記録のみを対象とする(差分同期。全件送信は行わない)
3. 送信先
   - Cloudflare Workers(写真AI判定と同じ中継サーバー)を経由してGoogle Sheets APIを呼び出し、**アプリのデータモデル専用に新規作成したスプレッドシート**(今使っている手動運用の既存スプレッドシートとは別物)の2タブに追記する
     - 体重記録タブ: `日付 | 体重(kg) | 体脂肪率(%) | メモ | 記録日時 | ID`
     - 食事記録タブ: `日時 | 区分 | 料理名 | カロリー(kcal) | たんぱく質(g) | 脂質(g) | 炭水化物(g) | ID`
     - 既存の手動運用シートは「1日1行、その日の食事の合計カロリー/PFC」という集計形式でアプリの`MealRecord`(1食事=1レコード)とは粒度が異なるため、あえて合わせずに新規シートとした
   - クライアントに認証情報を直接持たせないよう、認証(Googleサービスアカウント)はWorkers側で保持する
4. 成功時
   - 送信した記録の`synced`フラグを`true`に更新し、最終同期日時(`lastSyncedAt`)を更新する
5. 失敗時(オフライン、APIエラーなど)
   - `synced`フラグは`false`のまま維持し、記録自体は失わない
   - 次回のトリガー(アプリ起動時 or 手動)で再度同期を試みる
   - ユーザーには軽くエラーであることを伝える(派手なエラー画面は出さない)。設定画面のメッセージ表示以上の通知(バックオフ間隔・ホーム画面バッジ等)は追加しない(9章参照)

## 8. データモデル(案)

```
WeightRecord {
  id: string
  date: string (YYYY-MM-DD)   // 1日1件のキーとする
  timestamp: string (ISO8601) // 最後に保存した時刻
  weightKg: number
  bodyFatPercent?: number     // 体脂肪率(%)。任意入力、未計測日は未設定のまま
  note?: string
  synced: boolean             // スプレッドシートへの同期済みフラグ(7章参照)
}
// 同じdateで保存された場合は上書き(後勝ち)

MealRecord {
  id: string
  timestamp: string (ISO8601)
  mealType: "breakfast" | "lunch" | "dinner" | "snack"
  photoLocalRef?: string   // 端末内保存の画像参照(IndexedDB上のBlob等)。現時点では未使用(写真はAI判定にのみ使い、端末保存しない。4章参照)
  aiEstimatedName?: string
  aiEstimatedKcal?: number
  aiEstimatedProteinG?: number
  aiEstimatedFatG?: number
  aiEstimatedCarbsG?: number
  confirmedName: string
  confirmedKcal: number
  confirmedProteinG: number
  confirmedFatG: number
  confirmedCarbsG: number
  synced: boolean          // スプレッドシートへの同期済みフラグ(7章参照)
}

Settings {
  goalWeightKg: number
  goalDate: string (ISO8601 date)
  dailyCalorieTarget: number
  baselineDate?: string (YYYY-MM-DD)  // 進捗バーの起点日(6章参照)。未設定時は一番古い体重記録を起点とする
  lastSyncedAt?: string (ISO8601)  // 最終同期日時(6章の設定画面に表示)
}

FoodMasterItem {
  id: string
  name: string
  kcal: number
  proteinG: number
  fatG: number
  carbsG: number
  source?: string   // 出典(公式栄養成分表のURL等)。事前登録データの検証用、任意
  createdAt: string (ISO8601)
}
// 写真は持たない(名前・カロリー・PFCの数値のみ)。スプレッドシート同期の対象外(ローカルのみ)
```

## 9. 今後決めるべき事項

- カロリー計算(目標摂取カロリー)の自動計算ロジック(基礎代謝の考慮有無など)
- 食事写真を端末に保存するか(現状はAI判定に使うのみで保存していない。保存する場合は端末ストレージを圧迫しないためのリサイズ・圧縮方針も検討)
- 未同期件数・同期エラーの見せ方(ホーム画面でのバッジ表示、トースト通知など。同期本体はIssue #3)
- 推移画面でのPFC表示要否・表示形式

### 決定済みの論点(経緯の記録)

- AI判定結果画面と手動修正画面 → 1画面に統合(区分選択も含め、食事記録は1画面構成。4章参照)
- 体重の増減表示(前回比) → 体重入力欄の直下に表示する(3章参照)
- PFCの入力UI → 3つの数値入力欄を横に並べる(円グラフ的な比率表示は採用しない)
- 履歴確認画面の導線 → 推移画面内の「グラフ/履歴」サブタブ。期間フィルタは推移グラフと連動させず、全期間・日付降順の固定表示(5.1参照)
- 基準日ちょうどの体重記録が存在しない場合の進捗バー起点 → 一番古い体重記録にフォールバックする(6章参照)
