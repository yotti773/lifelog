# ライフログ収集・活用アプリ 要件定義書

作成日: 2026-07-01
最終更新日: 2026-07-06(フェーズ2: 日記機能の詳細仕様を追記。Issue #10)

## 1. 背景・目的

食事・水分・筋トレ・体重・日記など、日々のライフログを一元的に記録し、健康管理に活用するアプリを開発する。

- **短期目標**: 2026年10月31日までに体重を72kg → 64kg(-8kg)まで減量する
- **長期目標**: 健康管理に必要な情報をすべて一元的に扱えるプラットフォームにする

### 短期目標の数値的な裏付け
- 期間: 2026/7/1〜2026/10/31(約122日)
- 減量目標: -8kg
- 体脂肪1kgあたり約7,700kcalとして、総消費カロリー差は約61,600kcal
- 1日あたりの目安: **約505kcal/日の摂取抑制(または運動による消費増)**

## 2. 現状の運用(移行前の状態)

現在は以下のツールを組み合わせて手動でライフログ管理を行っている。

| ツール | 役割 |
|---|---|
| スプレッドシート | DB代わりにデータを蓄積 |
| LINE Bot | 記録の入力窓口 |
| あすけん | 食事・カロリー管理 |
| Garmin | 運動・活動量・(推定)睡眠等のトラッキング |
| NotebookLM | 上記データを読み込ませ、パーソナルトレーナー的な壁打ち相手として活用 |

→ 本アプリでは、これらに分散しているデータ入力・蓄積・振り返りのプロセスを一元化することを目指す。将来的には既存ツール(特にGarmin、あすけん)からのデータ連携・移行も検討余地あり(フェーズ2以降の論点、4章参照)。

## 3. MVPスコープ

期限(10月末)を踏まえ、減量目標に直結する機能を優先する。

| 優先度 | 機能 | 備考 |
|---|---|---|
| 最優先 | 体重記録 + 推移グラフ | 進捗確認の中核 |
| 最優先 | 食事記録 + カロリー・PFC計算 | 写真AI判定 + 手入力修正のセット構成。カロリーに加えPFC(たんぱく質・脂質・炭水化物)も記録する |
| 後回し | 筋トレ記録 | 減量目標への直接寄与は小さいため後続フェーズ |
| 後回し | 水分記録 | 同上 |
| 後回し | 日記 | 長期的な健康管理には必要だが短期目標には無関係 |

## 4. 機能要件

### 4.1 体重記録
- 日々の体重を記録
- 体重に加えて体脂肪率も記録対象とする(体組成計での計測値を想定、任意入力)
- 推移グラフを表示(期間切り替え: 週/月/全期間 など、詳細は今後詰める)。体重の推移グラフに加え、体脂肪率の推移グラフも表示する
- 推移グラフとは別に、記録した体重・体脂肪率を日付順の一覧で確認できる「履歴確認画面」を設ける(過去の記録を1件ずつ振り返る用途)
- 10月末64kg到達に向けた進捗表示(目標線・現在ペースでの着地予測など)を検討
- 設定画面で指定した「基準日」の体重記録を起点(0%地点)、目標体重を終点(100%地点)とした進捗バーを表示し、現在体重が基準日からどれだけ目標に近づいたかを視覚化する

### 4.2 食事記録・カロリー/PFC計算
- 入力方式は3通り:
  1. **写真を撮影またはライブラリから選択 → AI(Google Gemini APIのVision機能)が料理内容・推定カロリー・推定PFC(たんぱく質/脂質/炭水化物、g単位)を判定 → ユーザーが数値を手動で修正・確定**。判定時には「唐揚げ弁当、ご飯少なめ」のような補足情報(テキスト、任意)を添えることができ、写真だけでは判別しにくい料理名・分量・複数品目の内訳の判定に反映される
  2. **手入力**(料理名・カロリー・PFCを直接入力)
  3. **食事マスタから選択**(よく食べるものを事前登録しておき、選ぶだけでフォームに値をprefillする。あすけんの「選択式」入力に相当。選択後も通常の手動修正UIでその場で微調整できる)
- 単品(白米、りんご等)は精度が高いが、混合料理・外食メニューは誤差が出やすいため、修正前提のUI設計とする
- カロリーだけでなくPFCもAI判定・手動修正の対象とし、1日の合計値を記録する
- 1回の食事で複数品目(例: 弁当+サラダ+スープ)を記録する場合は、品目をリストに追加してまとめて保存できる(同じ区分に複数レコードとして保存される)
- 1日の摂取カロリー合計と、目標カロリー(505kcal/日抑制ベースの目安)との差分表示を検討
- PFCについては当面「実績の記録・表示」を優先し、目標バランスの設定要否は今後検討する(7章参照)

#### 食事マスタ(頻出品目の選択入力)
- 決まりきった品目(いつも同じ弁当・チェーン店メニューなど)を毎回撮影・入力し直す手間を減らすため、「よく食べるもの」を名前+カロリー・PFCの固定値として登録しておき、選ぶだけで記録できるようにする
- マスタの値は固定値をそのままprefillするのみとし、基準量に対する倍率計算などの数量調整機能は持たない(必要なら選択後に手動修正欄で都度調整する)
- マスタへの登録経路は2通り:
  1. 食事記録の保存時に「マスタに登録する」を選ぶと、その回の確定値(AI判定確定値 or 手入力値)がそのまま登録される
  2. 事前登録として、よく利用する外食チェーン・コンビニの定番メニューを、各社公式の栄養成分表を出典にシードデータとして登録する
- マスタには写真を持たせない(名前とカロリー・PFCの数値のみ)。食事記録本体の写真容量とは別の話であり、マスタ自体の容量は無視できる程度に抑える

### 4.3 データ同期(スプレッドシートへの書き出し)
- ローカル(IndexedDB)に保存された体重記録・食事記録それぞれに「同期済みかどうか」のフラグを持たせる
- オフライン中に記録した「未同期の記録」はローカルに蓄積しておき、消失させない(ローカルファースト方針を維持)
- 同期タイミング: (1)アプリ起動時にオンラインであれば自動的に未同期分をまとめて同期、(2)設定画面の手動同期ボタンでいつでも明示的に同期、の2通りを用意する
- 同期先はスプレッドシート(現状運用で使っていたスプレッドシートの後継)とし、アプリ→スプレッドシートの一方向書き込みとする(スプレッドシート側の編集をアプリに取り込む双方向同期は本MVPでは行わない)
- 同期失敗時は「未同期」のフラグを維持し、次回の同期タイミングで再試行する
- オフライン/オンラインの判定は `navigator.onLine` のみで行う(実際にリクエストを試みて判定する方式は採用しない。判定をすり抜けて同期に失敗した場合も未同期フラグが維持され次回再試行されるため、判定の厳密さは要求しない)

### 4.4 筋トレ記録(フェーズ2)
- **記録粒度はセット単位とする**: 種目ごとに「重量(kg)×回数」をセットごとに複数記録する(例: ベンチプレス 60kg×10回、60kg×8回、55kg×10回)。種目単位の合計値(総セット数のみ)ではなく、セットごとの数値を残すことで負荷の推移を正確に追える
- 1回のトレーニングで複数種目を記録できる。ただし1回の記録画面では1種目分(複数セット)を記録する構成とし、別の種目を続けて記録する場合はホーム画面経由で記録画面を再度開く(食事記録における「同じ区分で複数回記録してもよい」と同様の考え方。画面設計書5章参照)
- 種目の入力は、食事マスタと同様の**種目マスタ**(よく行う種目を事前登録し、選ぶだけで種目名をprefillする)+自由入力の併用とする。マスタは種目名のみを持ち、重量・回数はマスタ化せず毎回入力する(重量・回数はトレーニングのたびに変わる値のため、食事マスタのカロリー・PFCのような固定値prefillとは性質が異なる)
- ホーム画面に体重・食事と並べて当日の記録を表示する(画面設計書2章参照)。推移画面でのグラフ表示(重量推移など)はMVPでは行わず、まずは記録の蓄積と一覧確認を優先する(グラフ化は将来の拡張として別Issueで検討)
- 詳細仕様は画面設計書5章「筋トレ記録フロー」を参照

### 4.5 水分記録(フェーズ2)
- **記録単位はml**。値の入力は「コップ1杯(200ml)」「ペットボトル(500ml)」などのクイック追加ボタンをワンタップするだけで、その場で現在時刻の記録として即座に追加する(体重・食事のような専用の記録画面への遷移は行わない)。ボタンでカバーできない端数量は「手入力」から任意のml数・日時を指定して記録する
- 1日に複数回記録できる(クイック追加ボタンを何度も押す想定)。当日の合計mlがその日の摂取量になる
- 1日の目標摂取量(ml)を設定画面で設定でき、ホーム画面で当日合計と目標との差分(「あと◯◯ml」、目標達成時はその旨)を表示する(カロリーの目標差分表示と同様のパターン)。目標未設定の場合は差分を表示せず合計mlのみ表示する
- ホーム画面に体重・食事・筋トレと並べて当日合計mlを表示する
- 推移画面に日別摂取量の推移グラフを追加する(カロリー推移グラフと同様、記録の無い日も0mlで埋めて欠損を隙間として表示する)
- 記録の訂正は「削除して記録し直す」運用とし、個別記録の量を編集するUIは持たない(クイック記録主体のため、誤記録は削除→再度ワンタップで記録し直す方が手間が少ないと判断)
- 詳細仕様は画面設計書6章「水分記録フロー」を参照

### 4.6 日記(フェーズ2)
- **形式は自由記述 + 気分タグの簡易ハイブリッドとする**。本文は自由記述のテキストをメインとし、任意で気分タグ(3段階: 良い/普通/悪い)を添えられるようにする。体調・睡眠時間などの固定項目を埋めるテンプレート形式は採用しない(項目設計の手間を避け、まずは書く習慣自体を作ることを優先)
- **記録粒度は体重と同様、日付をキーとした1日1件とする**(後勝ちで上書き。同日に複数回保存した場合は最後の内容で上書きされる)
- ホーム画面に体重・食事・筋トレ・水分と並べて当日の日記(本文の冒頭を数十文字程度でプレビュー表示、未記録なら「未記録」)を表示する
- 写真添付はMVPでは行わない(テキストのみ)
- 詳細仕様は画面設計書7章「日記フロー」を参照

## 5. 非機能要件

### 5.1 利用環境
- スマホでの手軽な入力を最優先(Webアプリ/PWA)
- PCからの閲覧・確認は「たまに確認できれば十分」レベルで良い(リアルタイム同期は不要)

### 5.2 データ保存
- 基本はスマホ端末内(IndexedDB)に保存するローカルファースト方式とする
- PCでの確認やバックアップ用のJSONエクスポート/インポート機能は`src/db/backup.ts`にロジックとして実装済みだが、**設定画面のUIはMUI移行(Issue #27)時に削除した**。手動移行の主目的は後述のスプレッドシート同期でおおむね代替できるため。スプレッドシート側からの復元(インポート)はできない点は把握した上での判断であり、UIの再追加が必要になれば`db/backup.ts`をそのまま使って容易に戻せる
- 加えて、ネットワーク接続時には未同期の記録をスプレッドシートへ書き出す仕組みを持つ(4.3参照)。ただしこれは「常時同期のクラウドDB」ではなく、あくまで一方向のバックアップ・閲覧用途であり、ローカルファースト方針そのものは変えない
- 常時双方向同期のためのクラウドDB(Firebase, Supabase等)は本MVPでは採用しない(コストと運用の手間を最小化するため)
- 将来的な複数ユーザー対応(他者への配布)は現時点では未定・保留

### 5.3 AI活用
- 食事記録の画像判定にAI API(Google Gemini APIのVision機能)を利用する想定(既存で契約済みのAPIキーを活用するため)
- 将来的にはNotebookLMで行っているような「パーソナルトレーナー的な振り返り・アドバイス」機能をアプリ内に統合することも視野に入れる(フェーズ2以降)

## 6. 技術スタック

MVP開発においては「10月末までに動くものを作る」ことと「コストをかけない」ことを優先し、以下の構成を採用する。

| レイヤー | 技術 | 備考 |
|---|---|---|
| フロントエンド | React + Vite + PWA | スマホでの利用が主戦場 |
| データ保存 | ブラウザのIndexedDB(端末内保存) | サーバー・DBの構築・運用が不要になる。PC・スマホ間の要件はリアルタイム同期ではなく「たまに確認できれば十分」で合意済みのため、この方式を採用 |
| PC閲覧・バックアップ | JSONエクスポート/インポート機能を実装し、手動でデータを移す(ロジックのみ`db/backup.ts`に実装。設定画面のUIはIssue #27で削除。5.2参照) | クラウド同期基盤を持たずに複数端末での確認を実現する |
| 写真AI判定 | Cloudflare Workers(サーバーレス関数)を中継役として設置し、そこからGoogle Gemini API(Vision機能)を呼び出す | クライアントに直接APIキーを埋め込むと漏洩リスクがあるため、キーを隠すための最小限の中継が必要。フル機能のバックエンドは不要 |
| データ同期(スプレッドシート書き出し) | 写真AI判定と同じCloudflare Workersを中継役として使い、そこからGoogle Sheets APIを呼び出し、未同期レコードを追記する | 認証情報(サービスアカウント等)をクライアントに埋め込まないよう、既存のWorkers中継を流用する。アプリ→スプレッドシートの一方向書き込み |
| ホスティング(フロント) | Cloudflare Workers(Git連携ビルド + 静的アセット配信) | 当初はCloudflare Pagesを想定していたが、AI判定・同期の中継に使うWorkerと同一のWorker(`wrangler.toml` の `[assets]` ブロック)で静的配信までまとめる構成に変更。`main` へのpushで自動ビルド・デプロイされる |

### 検討の経緯・トレードオフ
- 当初はReact + Spring Bootによるフルスタック構成、およびクラウド同期(Firebase/Supabase等)を検討したが、以下の理由から上記のローカルファースト構成に変更した。
  - Render/RailwayなどのPaaS無料枠は、DBの自動削除やスリープ挙動があり、日次記録アプリの運用には不向き(2026年時点の調査結果より)
  - Oracle Cloud Always Freeによる自前サーバー運用はコストゼロだが、運用の手間・アカウント審査等の不確実性がある
  - 「PC閲覧はたまに確認できれば十分」という要件確認により、常時同期のためのサーバー・DBを持つ必要がなくなった
  - スプレッドシートへの同期は、2章に記載の「現状はスプレッドシートをDB代わりに使っている」運用を活かし、使い慣れたフォーマットでの閲覧・バックアップを実現するために採用した
- **Spring Boot(Java)の実務経験は、本MVPでは活用箇所がないが、フェーズ2(複数端末のリアルタイム同期、筋トレ・水分・日記機能の追加、NotebookLM的なパーソナルトレーナー機能の統合等)で本格的なバックエンドが必要になった際に活用する位置づけとする**

### この構成のメリット・留意点
- メリット: サーバー費用ゼロ、運用の手間がほぼゼロ、オフラインでも記録可能、開発速度が速い
- 留意点: 機種変更やブラウザデータ削除でローカルデータが消失するリスクがある。エクスポートUIを削除した現在は、スプレッドシート同期(一方向・復元不可)のみが off-device のコピーとなるため、リスクは完全には解消していない。定期的なエクスポートを促すUI・通知(Issue #14)は、エクスポートUI自体の要否と合わせて再検討する

### 決定済みとなった論点(経緯の記録)

- 技術スタックの選定 → 6章の構成で決定し、実装済み
- 画面ごとの詳細仕様 → 画面設計書として作成済み(未決部分は同書12章を参照)
- オフライン/オンラインの判定方法 → `navigator.onLine` のみで判定する(4.3参照)
- スプレッドシート同期の詳細設計(#3) → シートのフォーマット・列構成はアプリのデータモデルに沿った新規専用スプレッドシート(体重記録/食事記録の2タブ)、認証方式はGoogleサービスアカウント(Workers側のみが保持)、書き込みAPIはGoogle Sheets API v4の`values.append`で決定。既存の手動運用シートは1日1行の集計形式でアプリの`MealRecord`の粒度と合わないため、そちらへの合流は見送った(画面設計書10章参照)
- 同期失敗時のリトライ間隔・エラー通知の要否(#3) → 新しい仕組みは追加せず、既存の「未同期フラグを維持し次回起動時 or 手動ボタンで再試行」+ 設定画面でのエラーメッセージ表示のみで十分と判断
- 設定画面のJSONバックアップ(エクスポート/インポート)UI(#27) → MUI移行時にUIを削除。ロジック(`db/backup.ts`)は残置し、必要になれば再追加できる(5.2章、画面設計書9章参照)
- 筋トレ記録の記録粒度・種目管理・画面統合方針(#8) → セット単位(重量×回数をセットごとに記録)、種目マスタ+自由入力、ホーム画面に体重・食事と並べて表示、推移グラフはMVPでは持たず一覧確認のみで決定(4.4章、画面設計書5章参照)
- 水分記録の記録単位・目標設定・画面統合方針(#9) → ml単位、クイック追加ボタン+手入力、目標摂取量設定+ホームでの差分表示あり、ホーム画面に体重・食事・筋トレと並べて表示、推移グラフあり(カロリー推移と同様)で決定(4.5章、画面設計書6章参照)
- 日記の形式・記録粒度・画面統合方針(#10) → 自由記述+気分タグ(3段階)のハイブリッド形式、体重と同様に日付キーで1日1件(後勝ち)、ホーム画面に体重・食事・筋トレ・水分と並べて表示、写真添付はMVPでは行わない、で決定(4.6章、画面設計書7章参照)

## 8. まとめ

10月末までの減量目標達成を最優先とし、体重記録とAI活用の食事記録をMVPとして先行開発する。食事記録はカロリーに加えPFC(たんぱく質・脂質・炭水化物)も記録対象とし、より具体的な栄養バランスの把握に活用する。またローカルファースト方針を維持しつつ、ネットワーク接続時には未同期の記録をスプレッドシートへ同期する仕組みを備え、オフラインでも記録を止めない運用と、使い慣れたスプレッドシートでの確認を両立する。長期的には筋トレ・水分・日記を含めた包括的なライフログ管理基盤へと拡張し、現在スプレッドシート・LINE Bot・あすけん・Garmin・NotebookLMで分散運用している仕組みを本アプリに統合していく。
