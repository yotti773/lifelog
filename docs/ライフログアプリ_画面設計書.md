# ライフログアプリ 画面設計書(MVP)

作成日: 2026-07-01
関連ドキュメント: ライフログアプリ_要件定義書.md

## 1. 全体構成(ナビゲーション)

ボトムナビゲーション3タブ構成とする。

| タブ | 役割 |
|---|---|
| ホーム | 起動時に表示。その日の体重・食事記録を時系列でまとめて表示 |
| 推移 | 体重・カロリーのグラフ、目標までの進捗確認 |
| 設定 | 目標体重/目標日の設定、JSONエクスポート/インポート |

記録の入力は、ホーム画面右下の「＋」ボタン(フローティングアクションボタン)から「体重を記録」「食事を記録」をアクションシートで選択して行う。

## 2. ホーム画面

### 表示内容
- 当日の日付
- 体重(当日1件のみ表示。同日に複数回記録した場合は最後に保存した値で上書きされる=後勝ち)
- 食事記録一覧(朝食/昼食/夕食/間食の4区分。未記録の区分は「未記録」と表示)
- 当日の摂取カロリー合計、目標カロリーとの差分(「あと◯◯kcal食べられます」の形で表示)
- 当日の摂取PFC合計(たんぱく質/脂質/炭水化物、それぞれグラム表示)
- 各記録の同期状態(スプレッドシートへ未同期の場合は「未同期」のラベルを控えめに表示。7章の同期フロー参照)

### 画面イメージ
```
[7/1(水)]
─────────────
体重
 21:40  72.1kg
─────────────
食事
 朝食 07:30  トースト+卵    420kcal        [未同期]
 昼食 12:15  鶏肉と野菜炒め  580kcal
 夕食 --:--  (未記録)
 間食 --:--  (未記録)
─────────────
今日の摂取カロリー: 1,000kcal / 目標 1,900kcal
              (あと900kcal食べられます)
P: 45g  F: 30g  C: 120g

        [＋ 記録する]
```

## 3. 体重記録フロー

1. ホーム画面の「＋」→「体重を記録」を選択
2. 体重入力画面が開く
   - 日時: 現在時刻がデフォルト(タップで変更可能)
   - 体重: テンキー形式で入力(前回値からの増減も小さく表示すると励みになる)
   - メモ(任意): 「筋トレ後」「飲み会翌日」など一言メモ欄
3. 保存ボタンで確定 → ホーム画面に戻り、当日の一覧に反映される

- 1日1件のみ記録する仕様とする。同日に2回目以降を保存した場合は、その日の記録を新しい値で上書きする(後勝ち)

## 4. 食事記録フロー

1. ホーム画面の「＋」→「食事を記録」を選択
2. 区分選択画面: 朝食/昼食/夕食/間食から選択(現在時刻から一番近い区分をデフォルトで選択状態にしておく)
3. 写真撮影 or ライブラリから選択
4. AI判定結果表示画面
   - 判定された料理名(候補)
   - 推定カロリー(kcal)、推定PFC(たんぱく質/脂質/炭水化物、それぞれg)
   - 判定に自信が低い場合(混合料理など)は、その旨を軽く示すと良い(例: 「複数の料理が写っているため誤差が大きい場合があります」)
5. 手動修正画面(判定結果はここで必ず確認・編集可能)
   - 料理名の修正
   - カロリー数値の修正(タップして直接編集)
   - PFC(たんぱく質/脂質/炭水化物)数値の修正(それぞれタップして直接編集)
6. 保存ボタンで確定 → ホーム画面に戻り、当日の一覧に反映される

- 同じ区分(例: 昼食)に複数品目を記録したい場合は、同じ区分で複数回記録できるようにする(区分ごとの合計カロリー・PFCをホーム画面で集計)

## 5. 推移画面

- 体重推移グラフ
  - 1日1値(後勝ちで確定した値)を折れ線表示
  - 切り替え: 期間(週/月/全期間)
- カロリー推移グラフ(日別の摂取カロリー合計、目標ラインを重ねて表示)
- PFC推移の表示要否・表示形式(積み上げ棒グラフ等)は今後検討(9章参照)
- 目標までの進捗表示
  - 現在体重、目標体重(64kg)、残り日数(10/31まで)
  - 現在のペースで10月末に到達しそうな体重の予測(簡易線形予測)

## 6. 設定画面

- 目標体重・目標日付の設定(初期値: 64kg / 2026-10-31)
- 1日の目標摂取カロリーの設定(自動計算 or 手動上書き)
- JSONエクスポート(全データをファイル出力、PCでの確認・バックアップ用)
- JSONインポート(エクスポートしたファイルを読み込み、機種変更時などに復元)
- 定期エクスポートのリマインド通知(任意、忘れ防止)
- データ同期(スプレッドシート書き出し)
  - 「今すぐ同期」ボタン(手動同期のトリガー。押下時は同期中であることが分かるようローディング表示)
  - 最終同期日時の表示
  - 未同期件数の表示(任意、あれば同期の必要性が分かりやすい)
  - 同期失敗時はその旨をエラーメッセージとして表示し、次回同期のタイミングで自動的に再試行する

## 7. データ同期フロー

同期は「ローカルファースト」を維持したまま、ネットワーク接続時に未同期の記録をスプレッドシートへ書き出すための一方向(アプリ→スプレッドシート)の仕組みとする。

1. トリガー
   - アプリ起動時: オンラインであれば、起動直後にバックグラウンドで未同期レコードの同期を自動的に試みる(ユーザー操作は不要)
   - 手動: 設定画面の「今すぐ同期」ボタンから、いつでも明示的に同期を実行できる
2. 対象データ
   - `synced: false` の体重記録・食事記録のみを対象とする(差分同期。全件送信は行わない)
3. 送信先
   - Cloudflare Workers(写真AI判定と同じ中継サーバー)を経由してGoogle Sheets APIを呼び出し、スプレッドシートに追記する
   - クライアントに認証情報を直接持たせないよう、認証はWorkers側で保持する
4. 成功時
   - 送信した記録の`synced`フラグを`true`に更新し、最終同期日時(`lastSyncedAt`)を更新する
5. 失敗時(オフライン、APIエラーなど)
   - `synced`フラグは`false`のまま維持し、記録自体は失わない
   - 次回のトリガー(アプリ起動時 or 手動)で再度同期を試みる
   - ユーザーには軽くエラーであることを伝える(派手なエラー画面は出さない)

## 8. データモデル(案)

```
WeightRecord {
  id: string
  date: string (YYYY-MM-DD)   // 1日1件のキーとする
  timestamp: string (ISO8601) // 最後に保存した時刻
  weightKg: number
  note?: string
  synced: boolean             // スプレッドシートへの同期済みフラグ(7章参照)
}
// 同じdateで保存された場合は上書き(後勝ち)

MealRecord {
  id: string
  timestamp: string (ISO8601)
  mealType: "breakfast" | "lunch" | "dinner" | "snack"
  photoLocalRef?: string   // 端末内保存の画像参照(IndexedDB上のBlob等)
  aiEstimatedName?: string
  aiEstimatedKcal?: number
  aiEstimatedProteinG?: number
  aiEstimatedFatG?: number
  aiEstimatedCarbsG?: number
  confirmedName: string
  confirmedKcal: number
  confirmedProteinG: number
  confirmedFatG: number
  confirmedCarbsG: number
  synced: boolean          // スプレッドシートへの同期済みフラグ(7章参照)
}

Settings {
  goalWeightKg: number
  goalDate: string (ISO8601 date)
  dailyCalorieTarget: number
  lastSyncedAt?: string (ISO8601)  // 最終同期日時(6章の設定画面に表示)
}
```

## 9. 今後決めるべき事項

- AI判定結果画面と手動修正画面は、1画面にまとめるか2画面に分けるか(UI実装時に決定)
- 体重の増減表示(前回比)の要否・表示位置
- カロリー計算(目標摂取カロリー)の自動計算ロジック(基礎代謝の考慮有無など)
- 食事写真の保存容量(端末ストレージを圧迫しないよう、リサイズ・圧縮方針を検討)
- PFCの入力UI(3つの数値入力欄を並べるか、円グラフ的な比率表示にするか)
- 未同期件数・同期エラーの見せ方(バッジ表示、トースト通知など)
- 推移画面でのPFC表示要否・表示形式
