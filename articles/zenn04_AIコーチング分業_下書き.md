## はじめに

減量記録PWA「からだログ」には、週次レビュー画面に「AIコーチのコメント」欄があります。週の実績への総評・良かった点・来週の具体的アクションをAIが生成する機能です。

この機能の設計原則は1行で書けます。

**数値の計算・集計・安全判定はすべてコード側で決定論的に行い、AIには「計算済みの事実の解釈と言語化」だけをさせる。**

「記録データをAIに渡して分析してもらう」と聞くと、生レコードを全部プロンプトに詰めて「今週を評価して」と頼む構成を思い浮かべますが、そうしていません。この記事では、なぜ分業に倒したのか、そのために作った入出力の「契約」と、プロンプト・品質担保の実際を書きます。コードと設計書は公開しています: https://github.com/yotti773/lifelog

## なぜAIに計算をさせないか

理由は3つです。

**1. 軽量モデルで成立する。** 生レコードを渡して「分析して」と頼むと、高性能モデルでも計算を間違えることがあり、トークンも浪費します。計算済みダイジェストの言語化だけなら、最安クラスの軽量モデルで十分な品質が出ます。

**2. 数値の正しさが保証される。** 画面に表示する数値(週平均体重・必要ペースなど)はすべてコードが算出した値を直接表示し、AIの出力文は「解釈」の欄にだけ使います。AIが数字を言い間違えても、画面上の数値は常に正しい。

**3. 安全がAIの挙動に依存しない。** 「摂取カロリーが基礎代謝を割っている」「減量ペースが速すぎる」といった安全上の判定はコードが行い、結果をフラグとしてAIに渡します。AIの仕事は、それを踏まえた言い回しだけです。

要するに、LLMの得意な仕事(自然言語での言語化・励まし・優先順位づけ)と不得意な仕事(算術・一貫した閾値判定)を、アーキテクチャの段階で分けています。

## 入力の契約: WeeklyDigest

AIへの入力は`WeeklyDigest`という型のJSONだけです。コードが純関数として生成し、単体テストがあります。抜粋すると:

```typescript
interface WeeklyDigest {
  period: { start: string; end: string };  // 対象週(月曜〜日曜)
  weight: {
    weekAvgKg: number | null;              // 週平均体重
    weeklyChangeKg: number | null;         // 前週比(週平均同士の差)
    requiredWeeklyPaceKg: number;          // 必要ペース = 残り減量幅 ÷ 残り週数
    // ...
  };
  calories: {
    avgIntakeKcal: number | null;
    targetKcal: number;
    estimatedTdeeKcal: number | null;      // 実測TDEE(摂取と体重変化からの逆算)
    bmrKcal: number | null;
    // ...
  };
  recording: { recordedDays: number; currentStreakDays: number };
  flags: DigestFlag[];                     // コード側で判定済みの注意事項
  mood?: { good: number; normal: number; bad: number };  // 気分タグの件数集計
  // activity(Garmin)・workout・water などの週サマリーが続く。無い週はキー自体を省略
}
```

細かい工夫を3つ:

- **週平均同士で比較する。** 前週比は「日曜の体重 − 前の日曜の体重」ではなく週平均の差です。単日の体重は水分などのノイズが大きすぎるため
- **無いものはキーごと省略する。** Garmin未連携の週は`activity`キー自体が無い。「値が0」と「データが無い」をAIに区別させるため
- **画面とAIが同じ事実を見る。** 週次レビュー画面の表示にも同じ`WeeklyDigest`の値を使います。「画面ではこう見えているのにAIは別の数字を語る」という乖離が構造的に起きない

## 警告はコードが出す: DigestFlag

安全判定はすべてコード側の閾値判定です。例えば:

| フラグ | 判定条件(コード側で実装) |
|---|---|
| `PACE_TOO_AGGRESSIVE` | 週の減少幅が週平均体重の1%を超えている |
| `INTAKE_BELOW_BMR` | 週平均摂取カロリーが基礎代謝を下回っている |
| `LOW_RECORDING_RATE` | 記録した日が5日未満 |
| `INSUFFICIENT_DATA` | 記録した日が2日未満(評価に適さない週) |

そしてプロンプトで**「flagsに無い問題を新たに指摘しない」「flagsにある項目は必ず言及する」**の両方向を縛ります。AIに警告を発明させない、かつコードが検出した危険を握り潰させない。安全に関わる挙動を、確率的なモデル出力ではなく決定論的なコードとテストに載せるための要です。

## 出力の契約: WeeklyAdvice

出力も構造化JSONで強制します(Geminiのstructured output / `responseSchema`)。自由文をパースしません。

```typescript
interface WeeklyAdvice {
  verdict: "on_track" | "slightly_behind" | "behind" | "needs_attention";
  summary: string;      // 週の総評(2〜3文)
  wins: string[];       // 続けるべき良かった点(1〜2個)
  actions: string[];    // 来週の具体的行動(最大3個。測定可能な形で)
}
```

`verdict`はUIの色分けに直結し、`summary/wins/actions`は画面上のカード構造と1対1対応します。スキーマバリデーションを通らない出力はエラーとしてリトライ1回、それでもダメならAI欄だけ「生成に失敗しました」表示。**AI欄が失敗しても週次レビューの数値表示には一切影響しない** — これが分業の一番おいしい副産物です。AIはこのアプリで唯一の「壊れても本体が困らない」部品になっています。

## モデルとコスト: 月1円未満

モデルはGeminiのFlash-Lite系(その時点の最安モデル)です。入力はシステムプロンプト+digestで約2Kトークン、出力は約500トークン、実行は週1〜数回。**月のAPIコストは1円未満のオーダー**で、実質無料です。

これは分業設計の直接の帰結です。生レコードを渡す設計だと入力トークンは容易に数十倍になり、計算精度のために上位モデルが必要になります。「軽量モデルで足りる形に問題を変形してから渡す」のが、個人開発でLLMを常用機能に組み込むときの一番効くコスト戦略だと思います。

品質担保は大掛かりにしていません。代表的なdigestのフィクスチャを5〜6パターン(順調な週・停滞週・記録サボり週・ペース超過週・データ不足週)用意し、スキーマ通過とフラグ言及のキーワード検証を自動テストに、プロンプト変更時はフィクスチャ全パターンの目視確認を運用にしています。LLM-as-a-judgeのような評価基盤は、個人利用・週1実行の規模では過剰と判断しました。

## プライバシー: 日記本文はデフォルトで送らない

このアプリは日記も記録しますが、`WeeklyDigest`に入るのは気分タグの件数集計(good/normal/bad)だけで、**本文はデフォルトでは外部AIに送りません**。本文まで読ませたい人向けにオプトインのトグルを設け、ONの週だけ`diaryEntries`として含めます。画面の日記カードには、本文がAIに渡る状態かどうかを常に注記しています。

数値(体重・カロリー)と自由記述(日記)ではプライバシーの重さが違う、という線引きを設計の段階で固定しておくと、後から機能を足すときに毎回悩まなくて済みます。

## 同じ契約パターンは月次にもそのまま延びた

後から月次レビューを追加したときは、`MonthlyDigest`(月内各週の系列・月窓の実測TDEEなど、俯瞰に絞った項目)を作り、出力契約は週次と同じ`WeeklyAdvice`を流用しました。「digest型を定義 → コードで集計 → structured outputで受ける」という骨格が確立していたので、月次のAIコーチングはほぼ既存パターンの再演で済みました。入出力を契約として固定する設計は、機能追加の速度にも効きます。

## まとめ

- LLMに渡す前に、コードで計算を終わらせる。渡すのは小さな構造化ダイジェストだけ
- 安全判定はコードのフラグにし、プロンプトで「フラグに無い問題を指摘しない・あるものは必ず言及する」の双方向を縛る
- 出力はstructured outputで契約に固定する。AI欄が死んでも本体が動く構造にする
- 問題を軽量モデルで足りる形に変形すれば、常用AI機能が月1円未満で動く
- 生データとダイジェストの分離は、そのままプライバシーの線引き(日記本文はオプトイン)にもなる
