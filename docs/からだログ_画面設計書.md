## 1. 全体構成(ナビゲーション)

ボトムナビゲーション3タブ構成とする。

| タブ | 役割 |
|---|---|
| ホーム | 起動時に表示。その日の体重・食事・水分・日記・筋トレの記録をまとめて表示 |
| 推移 | 体重・カロリー・水分のグラフ、目標までの進捗確認、週次レビュー(8.2章)・月次レビュー(8.3章) |
| 設定 | 目標体重/目標日の設定、身体プロフィール、目標カロリー・PFC目標・目標水分摂取量、データ同期、食事マスタ・種目マスタ・習慣マスタ管理 |

記録の入力は、ホーム画面の各カード・ラベルを直接タップして対応する記録画面に遷移する形で行う(2章参照)。**フローティングアクションボタン+アクションシートによる「体重を記録」「食事を記録」の選択導線は、この直接タップ導線と役割が重複するためMUI移行(Issue #27)時に廃止した**(12章「決定済みの論点」参照)。

## 2. ホーム画面

### 表示内容
- 当日の日付
- 連続記録日数(フェーズ3、Issue #46)
  - 「記録した日」(その日に食事記録が1件以上、または体重記録がある日)が何日連続しているかを表示する。当日まだ未記録でも、前日まで連続していれば「継続中」として表示する(当日の未記録で即座に途切れた扱いにしない)
  - 常時表示のUIのため **accent色は使わない**(デザインガイドの制約。accentは達成の瞬間の演出専用)。日数の節目での達成演出を入れる場合のみaccentを使ってよい
- 体重・体脂肪率(当日1件のみ表示。同日に複数回記録した場合は最後に保存した値で上書きされる=後勝ち。体脂肪率は未記録なら非表示)
  - 体脂肪率も体重と同様、前回記録との差分(前回比 ±◯.◯%)をカード内に表示する。比較対象となる前回記録に体脂肪率が無い場合(前回未計測など)は「体組成計より」を表示する
  - 体重・体脂肪率カードはタップすると体重記録画面へ遷移する。当日分の記録が既にあればその値を編集、なければ新規入力になる(3章参照)
- 食事記録一覧(朝食/昼食/夕食/間食の4区分。未記録の区分は「未記録」と表示)
  - 同じ区分に複数品目がある場合は区分行に合計カロリーを表示し、品目ごとの内訳(時刻・品目名・カロリー)をその下にサブ行で表示する
  - 各区分行はタップ領域が2つに分かれる: アイコン+区分ラベル部分をタップすると、その区分をプリセットした状態で食事記録画面(新規)が開く。品目の詳細部分(時刻・料理名・カロリー)をタップすると、その品目の編集画面が開く。未記録の区分は行全体がタップ領域になり、区分をプリセットした状態で食事記録画面(新規)が開く(4章参照)
- 当日の摂取カロリー合計、目標カロリーとの差分(「あと◯◯kcal食べられます」、目標超過時は「◯◯kcalオーバーしています」の形で表示)
- 当日の摂取PFC合計(たんぱく質/脂質/炭水化物、それぞれグラム表示)
  - PFC目標値(9章参照)が設定されている場合は「実績g / 目標g」の対比表示にする。未設定の場合は従来どおり実績のみ(フェーズ3、Issue #47)
- その他の記録セクション(食事セクションの下に「水分」「日記」「筋トレ」「血圧」「周囲径」の行を表示。血圧・周囲径はフェーズ4、Issue #117・#118)
- 水分行
  - 当日合計ml(目標摂取量が設定済みの場合は「合計 / 目標ml」+進捗バー、未設定の場合は合計mlのみ)を表示する
  - タップすると水分記録画面へ遷移する。クイック追加・記録内訳の確認・削除はすべて記録画面側で行う(5章参照)
- 日記行(水分行の下)
  - 当日分の本文があれば冒頭を数十文字程度でプレビュー表示し、気分タグを設定していればラベルチップで併記する。未記録なら「未記録」と表示する
  - タップすると日記入力画面へ遷移する。当日分の記録が既にあればその内容を編集、なければ新規入力になる(6章参照)
- 筋トレ行(日記行の下)
  - 当日分の記録があれば「◯種目・◯セット」のサマリーを表示し、無ければ「未記録」と表示する
  - タップすると筋トレ記録画面へ遷移する。当日分の記録があればその内容がドラフトとして読み込まれ、追記・修正・削除がすべて記録画面側で行える(7章参照)
- 血圧行(フェーズ4、Issue #117)
  - 当日の測定値があれば「最高/最低 mmHg」を表示し、無ければ「未記録」と表示する。タップすると血圧記録画面へ遷移する(当日分があれば編集、なければ新規)
- サイズ(周囲径)行(フェーズ4、Issue #118)。**画面の表示ラベルは「サイズ(おなか周り)」**とし、測定用語の「周囲径」は避けて柔らかくする(記録画面のタイトルも「サイズを記録」、主フィールドの腹囲は「おなか周り」表記。内部の型名 `BodyMeasurementRecord`・同期シートのタブ名「周囲径記録」は変えない)
  - 最新のおなか周り(cm)と測定日を表示し、記録が無ければ「未記録」と表示する(月1程度の低頻度入力のため当日でなくても最新値を出す)。タップすると記録画面へ遷移する
- 今日の習慣セクション(その他の記録の下。フェーズ4、Issue #113)
  - アクティブな習慣をチェックリスト形式で表示し、タップで当日のチェックをON/OFFする。習慣ごとに連続日数・直近7日の達成状況を併記する(#46の仕組みを流用)。アクティブな習慣が無ければセクションごと非表示(管理は設定画面の習慣マスタ)
- 各記録の同期状態(スプレッドシートへ未同期の場合は「未同期」のラベルを控えめに表示。10章の同期フロー参照)

## 3. 体重記録フロー

1. ホーム画面の体重・体脂肪率カードをタップする(当日分の記録が既にあればその値を編集、なければ新規入力)
2. 体重入力画面が開く
   - 日時: 現在時刻がデフォルト(タップで変更可能)
   - 体重: 数値入力欄で入力(スマホでは数値キーボードが開く)。入力すると前回記録との差分(前回比 ±◯.◯kg)を入力欄の下に小さく表示する
   - 体脂肪率(任意): 体重と同様の数値入力欄。体組成計での計測値がない日は未入力のままでよい
   - メモ(任意): 「筋トレ後」「飲み会翌日」など一言メモ欄
3. 保存ボタンで確定 → ホーム画面に戻り、当日の一覧に反映される

- 1日1件のみ記録する仕様とする。同日に2回目以降を保存した場合は、その日の記録を新しい値で上書きする(後勝ち)

## 4. 食事記録フロー

記録は1画面で完結する(区分選択・AI判定結果・手動修正を別画面に分けない。12章にあった「1画面にまとめるか2画面に分けるか」の論点は1画面統合で決定済み)。

1. ホーム画面の食事区分ラベル(朝食/昼食/夕食/間食のアイコン+ラベル部分、未記録の区分は行全体)をタップする
2. 食事記録画面(1画面)で以下を入力する:
   - 区分: 手順1でタップした区分をデフォルトで選択状態にしておく(現在時刻から一番近い区分を選ぶロジックは、ホーム経由ではなく食事記録画面に直接アクセスした場合のフォールバックとして残す)
   - 日時: 現在時刻がデフォルト(変更可能)
   - 入力フォーム: 料理名・カロリー・PFC(たんぱく質/脂質/炭水化物)を直接入力・修正する欄。写真判定・マスタ選択の結果もこのフォームに反映されるため、どの入力方式でも保存前にここで必ず確認・修正できる
3. 手入力の代わりに、同じ画面内の以下のセクションからフォームへ値を入れることもできる(画面上の並び順は 手入力フォーム → 写真 → マスタ):
   - **写真から記録する**: 補足情報(任意のテキスト。例: 「唐揚げ弁当、ご飯少なめ」)を添えて、撮影またはライブラリから写真を選択 → AIが写っている料理・食品を品目ごとに個別に判定する(Issue #50)。写真は最大4枚まで添付でき(ライブラリは複数選択可、撮影は1枚ずつ追加。サムネイル一覧から1枚ずつ取り消し可能)、複数枚はすべて「同じ1回の食事」の写真として1回の解析にまとめて送られる — 品数が多い食事や後から追加した一品を、解析を分けずに記録するため(Issue #110)。判定結果が1品のみの場合は料理名・カロリー・PFCがフォームに直接prefillされる。複数品目を検出した場合は「検出した品目」一覧が表示され、タップでフォームに反映→内容を確認・修正→「この品目を追加してもう1品記録」でリストに追加、を品目ごとに繰り返す(自動で一括登録はしない。誤検出の確認・修正を挟むため)。未追加の検出品目が残ったまま保存しようとした場合は、記録漏れでないかの確認ダイアログを表示する。量や内容の判定の自信が低い場合は、その旨の注意文言を表示する
   - **よく食べるものから選ぶ**: 食事マスタを品目名で検索でき、一覧はページネーション(6件/ページ)で表示する。選ぶと登録済みの固定値がフォームにprefillされる
4. 複数品目をまとめて記録する場合は「この品目をリストに追加してもう1品記録する」でリストに溜めてから、最後にまとめて保存する(同じ区分・同じ日時の複数レコードとして保存され、区分ごとの合計カロリー・PFCはホーム画面で集計される)
5. 「この内容をマスタに登録する」チェックボックス(オンで保存時に食事マスタへ追加登録される)
6. 保存ボタンで確定 → ホーム画面に戻り、当日の一覧に反映される

- 同じ区分(例: 昼食)に複数品目を記録したい場合は、上記4のまとめて保存のほか、同じ区分で複数回記録してもよい
- 写真はAI判定への送信時に1枚ずつ長辺1024pxへリサイズ・JPEG圧縮され、判定にのみ使われる。**現時点では写真自体を端末に保存しない**(保存するかどうかは12章の論点)
- ホーム画面の食事品目(単一品目の行、複数品目時は各サブ行)をタップすると編集画面に遷移する(`?id=<MealRecord.id>` で食事記録画面を編集モードにする。区分・日時・料理名・カロリー・PFCを事前入力表示する)。編集フォーム本体は編集対象レコードを指すが、それとは別に**同じ区分・日時の新しい記録として品目を追加**することもできる。写真判定・マスタ選択・手入力のいずれからでも「追加で記録する品目」リストに積め、更新時に本体の上書きと合わせて新規レコードとして保存される(写真判定・マスタ選択からの追加はIssue #71、手入力での追加はIssue #99)
- 編集画面では「更新する」ボタンに加え「この記録を削除する」ボタンを設ける。削除は復元不可な操作のため、確認ダイアログ(削除対象の内容を表示し、キャンセル/削除するの2択)を1枚挟む
- 更新すると同期状態は`synced: false`にリセットされ(体重編集と同様)、次回の同期対象になる。更新・削除後はホーム画面に戻る(Issue #24)

### 4.1 食事マスタ管理

- 食事マスタ(登録済みの「よく食べるもの」)の一覧・編集・削除は、設定画面から遷移する専用画面(食事マスタ管理画面)で行う(9章参照)
- 一覧の各項目は名前・カロリー・PFCを表示し、その場でインライン編集または削除ができる
- 品目名での検索・ページネーションを備える
- 一括登録は同期用スプレッドシートの「食事マスタ」タブへ行を直接書き込み、「シートから取り込み」(10.1章参照)を実行する運用とする(Issue #96)。かつてコードに埋め込んでいた定番メニューのシードデータ(外食チェーン・コンビニの定番。「定番メニューを一括登録」ボタンで投入)は、シート貼り付け用のサンプル `docs/食事マスタ_定番メニューサンプル.csv` へ退避して撤去した
- 手動での新規追加(名前・カロリー・PFCを直接入力して1件登録)ボタン(一覧下部の破線ボーダーのボタン。Issue #35)
- 重複登録(似た名前・同じ商品が複数件になる)の自動マージは行わず、気づいたユーザーが手動で整理する想定とする

## 5. 水分記録フロー

ホーム画面の水分行をタップして専用の記録画面(水分を記録)へ遷移し、クイックタップで当日の摂取量を積み上げる構成とする(2章参照)。フェーズ2で詳細を決定(Issue #9。当初はホーム画面内でのワンタップ完結+手入力モーダルの構成としていたが、デザインプロトタイプの確認により専用画面+4ボタン構成に変更。意思決定ログ参照)。

1. ホーム画面の水分セクション(2章参照)をタップすると水分記録画面が開く
2. 画面上部に当日の合計カード(当日合計ml / 目標ml + 進捗バー)を表示する
   - 目標摂取量(9章参照)が未設定の間は進捗バーを表示せず、「設定画面で目標を設定すると、達成度を表示できます」の案内文を表示する
3. クイック追加ボタン(200 / 300 / 500 / 2000ml の4ボタン。当初は100/200/350/500mlで500mlをプライマリ強調としていたが、Issue #79で実際の飲用シーンに合わせて量を変更し、強調は廃止して全ボタン白抜きに統一)をタップすると、タップした瞬間の時刻・その量で記録が1件追加される(確認ダイアログなし。誤タップした場合は後述の記録リストから削除する)
   - 任意量の手入力は設けない(端数はボタンの組み合わせで近似する。ボタン量のカスタマイズ可否と合わせて12章の論点)
4. 合計カードの下に「今日の記録」リスト(時刻・量、新しい記録が上)を表示し、各行に削除ボタンを設ける。誤って記録した場合は削除し、必要なら正しい量で記録し直す(量そのものを編集するUIは持たない。クイック記録主体のため、削除→再記録の方が手間が少ないと判断)
   - 記録が0件の場合は空状態文言(「まだ記録がありません」)を表示する

- 1日の目標摂取量は設定画面(9章参照)で設定する(ml、手動入力、初期値未設定)。設定済みの場合のみホーム行・記録画面に進捗バーを表示する
- 推移画面(8章参照)に日別摂取量の推移グラフを追加する。カロリー推移グラフと同様、記録の無い日も0mlで埋めて欠損を隙間として表示し、目標ラインがあれば重ねて表示する
- 水分関連のUIはコーラル系ではなくブルー系(#3AA0DB / #2B7BB0 など)の配色を使う(水を想起させるため。デザインプロトタイプ参照)
- 履歴確認画面(8.1章)への統合は実装済み(#73。日付ごとに集約した1日1行で表示)
- スプレッドシート同期の対象に含める(#72で決定・実装済み。10章参照)。体重・食事と同様の削除トゥームストーンの仕組みを流用する
- クイック追加ボタンの量(200/300/500/2000ml)を設定でカスタマイズ可能にするかは今後検討(12章参照)。現状は固定値とする

## 6. 日記フロー

体重と同様、日付をキーとした1日1件の記録とし、ホーム画面からの直接タップで記録画面に遷移する構成とする(2章参照)。フェーズ2で詳細を決定(Issue #10)。

1. ホーム画面の日記セクションをタップする(当日分の記録が既にあればその内容を編集、なければ新規入力)
2. 日記入力画面が開く
   - 日付: 当日固定(体重と同様、日付が記録のキーになるため過去日の新規作成はこの画面からは行わない)
   - 気分タグ(任意): 「絶好調 / 良い / 普通 / 眠い / 不調」の5段階から1つ選択できるタイル(手描きストローク調の表情アイコン+ラベル。選択中はアンバー枠。当初は3段階の想定だったが、デザインプロトタイプの確認により5段階に変更。意思決定ログ参照)。テンプレート項目(体調・睡眠時間など)は持たない
   - 飲酒タグ(任意): 「飲酒あり」のトグル1つ(#112)。気分タグと同列の任意入力で、週次レビューのクロス分析(8.2章)の集計軸に使う。ONで保存した日だけ`alcohol: true`を持ち、OFFは「飲酒なし」ではなく「記録なし」(undefined)として扱う。飲酒は食事記録にカロリーとして載っていても「飲んだ日」というフラグにはならないため、1日単位の事実として日記側に持たせる(意思決定ログ参照)
   - 本文: 自由記述のテキストエリア(任意。「ひとこと日記」)
3. 下部固定の保存ボタンで確定 → ホーム画面に戻り、当日のプレビューに反映される

- 1日1件のみ記録する仕様とする。同日に2回目以降を保存した場合は、その日の記録を新しい内容で上書きする(後勝ち。体重記録と同じ考え方)
- 本文・気分タグ・飲酒タグのすべてが空の状態で保存した場合は、その日の記録を削除する(未記録に戻す)。この挙動により明示的な削除ボタンは設けない
- 写真添付はMVPでは行わない(テキストと気分タグのみ)
- 更新すると同期状態は`synced: false`にリセットされ、次回の同期対象になる(#72で同期対象に追加済み。10章参照)
- 履歴確認画面(8.1章)への統合・過去分の編集導線は実装済み(#73。履歴の日記行タップで`?date=YYYY-MM-DD`付きの日記入力画面を開き、過去日を編集できる)

## 7. 筋トレ記録フロー

ホーム画面の筋トレ行をタップして記録画面(筋トレを記録)へ遷移し、当日の全種目・全セットを1画面でまとめて編集する構成とする(2章参照)。フェーズ2で詳細を決定(Issue #8。当初は「1画面1種目+セット逐次確定」の構成としていたが、デザインプロトタイプの確認により「1画面で当日の複数種目をまとめて編集」に変更。意思決定ログ参照)。

1. ホーム画面の筋トレセクション(2章参照)をタップすると筋トレ記録画面が開く
   - 当日分の記録が既にあればその内容がドラフトとして読み込まれ、追記・修正・削除がすべてこの画面で行える(新規/編集で画面を分けない)
   - 当日分が無ければ、空の種目カード1枚(名前空欄+空セット1行)の状態で開く
2. 種目カードを縦に並べて編集する。各カードの構成:
   - 種目番号バッジ+種目名入力欄(自由入力。種目マスタ(7.1章参照)に登録済みの種目名は入力候補としてサジェスト表示される)+種目削除ボタン
   - 種目名が過去の記録にある種目と一致した場合、種目名欄の下に「前回 M/D + 前回の重量×回数」の参照行を表示し、「入力」ボタンでその内容をセット欄へ一括転記できる(Issue #100。下記の入力補助の項参照)
   - セット行のリスト(セット番号+重量(kg)・回数の数値入力欄+行削除ボタン)
   - 「セットを追加」ボタン(破線)で空のセット行を追加する
3. 末尾の「種目を追加」ボタン(破線)で種目カードを追加する
4. 下部固定の保存ボタンで当日分をまとめて確定する → ホーム画面に戻り、当日のサマリーに反映される
   - 保存時、重量・回数がどちらも空のセット行と、名前もセットも空の種目カードは除外する
   - 保存は「当日分の置き換え」として動作する(既存の当日レコードを消して保存内容で登録し直す)。すべての種目を削除して保存すれば当日分の削除になるため、明示的な削除ボタンは設けない

- 記録粒度はセット単位(1セット=1レコード。種目名・セット番号・重量kg・回数を持つ。11章参照)。記録日時は保存時刻を全レコード共通で持つ
- 種目マスタは種目名のみを持ち、重量・回数はマスタ化しない(毎回変わる値のため)
- 前回記録値は「自動prefill」ではなく「参照表示+任意の転記」として持つ(Issue #100): 種目名が過去の記録と一致すると、その種目を最後に記録した日の重量×回数を参照表示し、ユーザーが「入力」ボタンを押したときだけセット欄へ転記する。開いた瞬間に勝手に埋めないことで記録画面をシンプルに保ちつつ、繰り返し種目の再入力の手間を減らす。比較対象は編集中の日より前の最新記録(当日は含めない)
- メモ欄・保存時のマスタ自動登録は持たない(記録画面をシンプルに保つ。要否は12章の論点として再検討)
- 保存すると当日分の各セットレコードは`synced: false`で登録し直され、次回の同期対象になる(#72で同期対象に追加済み。置き換え保存に伴うID振り直し・削除トゥームストーンの扱いは10章参照)
- 推移画面でのグラフ表示(重量推移など)はMVPでは行わない。まずは記録の蓄積とホーム画面での一覧確認を優先し、グラフ化の要否は12章の論点とする
- 履歴確認画面(8.1章)への統合は実装済み(#73)。推移グラフへの統合は引き続き12章の論点

### 8.2 週次レビュー画面(フェーズ3)

※実装状況: 実装済み(Issue #45、AIコメント欄はIssue #12。2026-07-10)。画面は `src/pages/trends/WeeklyReview.tsx`(AIコメント欄は同 `WeeklyAdviceCard.tsx`)、集計は `src/db/weeklyReview.ts` → `src/lib/weeklyDigest.ts`。

- 推移画面の「グラフ/履歴/レビュー」サブタブの3つ目として実装する(Issue #45)。**新規画面のため、実装前にArtifactでHTMLモックアップを作成してレイアウトを確認する**(CLAUDE.mdの開発フロー)
- 週の定義は**月曜〜日曜**。デフォルトは今週を表示し、「◀ / ▶」で前週・次週に切り替えられる
- 週次データの集計は `WeeklyDigest`(11章・AIコンサルティング設計書参照)を生成する純関数として実装し、画面はその値を表示するだけにする。`WeeklyDigest` はAIコーチング(Issue #12)への入力契約と共通
- 表示セクション(依存Issueが未実装・データ不足のセクションは非表示にして段階的にリリースできる構成とする):
  - **体重**: 週平均体重、前週比(週平均同士の比較。単日の値は水分等のノイズが大きいため使わない)、目標までの必要ペース(kg/週)との比較
  - **カロリー**: 週の平均摂取kcal / 目標kcal、目標達成日数(◯/7日)
  - **記録率**: 記録した日数/7日(Issue #46)
  - **PFC平均**: 目標値(9章)が設定されていれば「実績 / 目標」の対比表示(Issue #47)
  - **活動(Garmin)**: 週平均歩数・週平均総消費カロリー・平均睡眠時間・活動データがある日数(Issue #82)。週内に活動記録が無ければセクションごと非表示(Garmin未連携ユーザーに影響しない)
  - **血圧(フェーズ4、Issue #117)**: 週平均(最高/最低)・記録した日数・家庭血圧135/85以上だった日数・同じ週の週平均体重(体重×血圧を並べて見せる)。週内に血圧記録が無ければセクションごと非表示。医療機器ではないため事実の提示に留め医学的判断はしない旨を注記する。この `bloodPressure` セクションはAIコーチング入力(WeeklyDigest)にも含まれ、AIにも同じ免責(血圧値の医学的評価・受診勧奨を禁止)を課す
  - **筋トレ**(Issue #103): 記録した日数・種目数・総セット数。週内に筋トレ記録が無ければセクションごと非表示
  - **水分**(Issue #103): 記録がある日の平均摂取量(記録の無い日は分母に入れない)、目標達成日数(目標未設定なら非表示)、記録した日数。週内に水分記録が無ければセクションごと非表示
  - **この週の日記**(Issue #103): 週内の日記(日付・気分タグ・飲酒タグ・本文)をローカル表示する(digestとは独立に取得。表示自体はAI送信オプトインと無関係)。カード下部に、本文がAIコメント生成に渡る状態かどうか(設定「日記の本文をAIに送る」のON/OFF)を常に注記する。日記が無い週は非表示
  - **実測消費カロリー(実測TDEE)**: 逆算値と、それに基づく目標カロリーの補正提案(Issue #44。有効週が無い間は「データ蓄積中」表示)。Garminの週平均総消費があれば併記し、両者の乖離が15%を超える週は「食事の記録漏れ等の可能性」の注記を出す(独立した2つの推定値の一致度を記録の信頼度の目安にする。Issue #82)
  - **クロス分析**(Issue #112): 週内データの掛け合わせ集計(`WeeklyDigest.crossAnalysis`)を事実の提示スタイルで表示する。週単位はサンプル数が少なく「相関」とは言い切れないため、断定・解釈はせず「睡眠6時間未満の日が2日あり、その日の平均摂取カロリーは2,050kcal(それ以外の日は1,770kcal)でした」のような事実文に留め、解釈はAIコメントに委ねる。カード下部に「日数が少ないため差は偶然の範囲のこともある」旨を常に注記する。項目は3つで、比較が成立する項目だけ表示し、1つも無い週はカードごと非表示:
    - **睡眠×食事**: 睡眠6時間未満の日とそれ以外の日の、同じ日の摂取カロリー比較。Garminの睡眠時間は「その日の朝までの夜間睡眠」のため、同日ペアリングが「睡眠不足明けの日の食事」を意味する(翌日ペアリングにしない。意思決定ログ参照)。睡眠不足の日に食事記録が無い週は非表示
    - **気分×食事**: 気分が良い日(絶好調・良い)と眠い・不調の日の摂取カロリー比較。両群に食事記録のある日が無い週は非表示
    - **飲酒×食事**: 飲酒タグ(6章)のある日の日数と、当日・それ以外の日・翌日(週内のみ)の平均摂取カロリー。「それ以外の日」はタグの無い日全体であり「飲酒なしと記録した日」ではない。飲酒日が無い週は非表示
  - **警告フラグ**: ペースが速すぎる(週あたり体重の1%超)、平均摂取が基礎代謝未満、記録率の低下など。コード側で判定した結果を表示する(AIの判断に依存しない)
  - **AIコーチのコメント**(Issue #12、このセクションのみAIを使う): 「生成する」ボタンで週の総評・良かった点・来週のアクションを生成し、`AdviceRecord`(11章)としてキャッシュする。生成済みの週はキャッシュを表示し、再生成ボタンを設ける

**維持モード(#116。未実装。目標タイプ=維持のとき)**: 週次レビューの評価軸を「減量ペースに乗っているか」から「維持レンジ内に収まっているか」へ切り替える(設定の目標タイプは9章)。表示・判定の差分:
- **体重**セクション: 「必要ペース(残りN日)」「現在ペースでの着地予測」を、「維持レンジ(例: 63.0〜65.0kg)」と「週平均体重がレンジ内か / レンジからの乖離(上抜け・下抜けと何kgか)」の表示に置き換える(`WeeklyDigest.weight` の `withinRange`/`rangeDeviationKg`。AIコンサルティング設計書3章)
- **実測消費カロリー**セクション: TDEE補正提案を「維持カロリー = 消費カロリー(赤字なし)」の提案に切り替える(設定の自動計算と同じ `suggestCalorieTarget()` を維持モードで呼ぶ。9章・要件定義書4.7.7章)

### 8.3 月次レビュー画面(フェーズ3)

※実装状況: 実装済み(Issue #114。2026-07-14)。画面は `src/pages/trends/MonthlyReview.tsx`(AIコメント欄は同 `MonthlyAdviceCard.tsx`)、集計は `src/db/monthlyReview.ts` → `src/lib/monthlyDigest.ts`。クロス分析カードは週次と共有(`CrossAnalysisCard.tsx`)。

- 週次レビュー(8.2章)は「今週の達成→来週のアクション」という行動を変えるレビューとして機能する一方、ペースの加速・減速や目標までの残距離といった**高度を上げた俯瞰**は週の粒度では見えない。そこで月次ならではの内容に絞った月次レビューを追加する(Issue #114)。**週次のカードを30日窓で並べ直すのではなく、内容を月次向けに絞る**
- **UIはサブタブを増やさず、レビュータブ内に「週/月」の切り替え**(`SegmentedControl`)を置く。「週」で週次レビュー(8.2章)、「月」で月次レビューを表示する。デフォルトは「週」。月ナビゲーション「◀ / ▶」で前月・翌月に切り替えられる(今月表示中は翌月へ進めない)
- **月の定義**: 暦月は月曜始まりの週と端が揃わないため、**「その月に日曜(週の終わり)が含まれる月曜始まりの週の集合」**とする(4〜5週)。週を分割せず日曜の属する月へ丸ごと割り当てるため、全ての週がちょうど1つの月に属する(隣接月と重複・欠落しない)。例: 2026年7月は6/29週(日曜7/5)〜7/20週(日曜7/26)の4週で、期間は6/29〜7/26。7/27週(日曜8/2)は8月に属する。実装は `src/lib/date.ts` の `weekStartsOfMonth`(経緯は意思決定ログ)
- 月次データの集計は `MonthlyDigest`(11章・AIコンサルティング設計書参照)を生成する純関数として実装し、画面はその値を表示するだけにする。週次の集計関数(週次TDEE逆算・クロス集計)を再利用する。`MonthlyDigest` はAIコーチング(月次版)への入力契約と共通
- 表示セクション(データ不足のセクションは非表示にして段階的に見せる):
  - **体重(週平均の推移)**: 月内の週平均体重を折れ線(4〜5点)で表示し、ペースの加速・減速を見せる。単日ではなく週平均同士を比較する8.2章の設計思想を延長する。折れ線は手書きSVG(他グラフと同じ方針)で、記録が無い週は点を打たず隙間にする。月間変化(記録がある最初の週と最後の週の週平均の差)・平均ペース(kg/週)・体重を記録した週数を併記する
  - **目標へのマイルストーン**: 目標日までの残り期間、必要ペース(kg/週)、**今月のペースを維持した場合の目標日時点の到達見込み体重**。見込みが目標を上回る/下回るで順調・要注意を色分けする
    - **維持モード(#116。未実装。目標タイプ=維持のとき)**: このセクションを「維持レンジ内安定度」に置き換える。必要ペース・到達見込みは出さず、月内の各週の週平均体重がレンジ内に収まっているか、複数週にわたる上抜けの継続(リバウンド傾向)が無いかを見せる。維持期は月単位の安定監視が主役になるため、月次レビューが維持期の中心ビューになる(要件定義書4.7.7章・AIコンサルティング設計書10章)
  - **実測消費カロリー(月間推定)**: 月窓での実測TDEE。週単位の逆算はブレが大きいため、月内の全有効週の週次逆算値を平均した安定値を出し、週ごとの逆算値の幅(最小〜最大)も併記する。目標カロリー補正提案(#44)の信頼度を上げる材料になる(有効週が無い間は「データ蓄積中」表示)
  - **カロリー・記録**: 月内の平均摂取kcal / 目標、目標以内の日数(◯/月の総日数)、食事を記録した日数、記録率(体重または食事)
  - **血圧(月次。フェーズ4、Issue #117)**: 月平均(最高/最低)・記録した日数・家庭血圧135/85以上だった日数。週内に記録が無ければ非表示。週次と同じく事実の提示に留め医学的判断はしない。月窓は週次よりサンプルが多く、体重×血圧の傾向が安定して見える(月次では週平均体重の併記はせず、体重の推移は上部の体重セクションで見せる)
  - **クロス分析(月間)**: #112の集計を月窓で再実行する(`CrossAnalysisCard` を週次と共有)。週次より標本数が多いため、注記は「週次より日数が多いぶん傾向の目安になるが、因果の断定はできない」旨に変える。項目(睡眠×食事・気分×食事・飲酒×食事)と非表示条件は週次と同じ

## 10. データ同期フロー

同期は「ローカルファースト」を維持したまま、ネットワーク接続時にローカルの変更(新規・編集・削除)をスプレッドシートへ書き出すための一方向(アプリ→スプレッドシート)の仕組みとする。これに加えて、シート→アプリ方向の**手動取り込み(インポート)**を持つ(Issue #54。本章末尾参照)。常時双方向同期にはしない。

※実装状況: 実装済み(Issue #3、削除・編集の反映はIssue #30、取り込みはIssue #54、水分・筋トレ・日記の追加はIssue #72、食事マスタ・種目マスタの追加はIssue #96)。クライアント側の同期エンジン(未同期レコード・削除トゥームストーンの抽出、部分成功のハンドリング、失敗時の未同期維持、起動時/手動のトリガー)に加え、送信先となるWorker→Google Sheetsのトランスポート(`workerSheetsTransport` → `POST /api/sync-sheets`)も実装済み。

1. トリガー
   - アプリ起動時: オンライン(`navigator.onLine` で判定)であれば、起動直後にバックグラウンドで未同期レコードの同期を自動的に試みる(ユーザー操作は不要)
   - アプリ復帰時・オンライン復帰時(Issue #105): PWAをホーム画面から開き直したときの再表示(`visibilitychange`)と、オフライン→オンライン復帰(`online`イベント)でも同期を試みる。タブ切り替え等での連続発火を防ぐため自動トリガーには最小間隔(5分)のスロットリングと実行中の再入抑止を設ける(`src/sync/autoSync.ts`)。オンライン復帰は「同期が可能になった」明確なシグナルのため最小間隔を待たずに実行し、オフラインでスキップした試行は間隔に数えない
   - 手動: 設定画面の「今すぐ同期」ボタンから、いつでも明示的に同期を実行できる(スロットリングの対象外)
2. 対象データ
   - `synced: false` の体重記録・食事記録・水分記録・筋トレ記録(セットレコード)・日記記録・食事マスタ・種目マスタ・血圧記録・周囲径記録・習慣マスタ・習慣記録(新規・編集分)を対象とする(差分同期。全件送信は行わない)。マスタも同期対象とするのは、記録と違いバックアップ手段が他に無く、シート直接編集による一括登録の受け口を兼ねるため(Issue #96。#14の残課題を引き継いだもの)。血圧・周囲径・習慣はフェーズ4の追加(Issue #117・#118・#113)
   - 加えて、同期済みの可能性がある記録を削除した際に残す**削除トゥームストーン**(`syncDeletions` テーブル。削除対象のタブとID列の値を保持)も対象とする。編集は`synced`が`false`に戻ることで、削除はトゥームストーンによって、それぞれ次回同期で拾われる(Issue #30)
   - 筋トレは「その日の全セットをまるごと置き換え」という保存方式(7章参照)のため、置き換えのたびに全セットへ新しいIDを振り直す。そのため置き換え前の全セットIDを削除トゥームストーンとして残し、シート側の古いセット行を次回同期で消す(Issue #72)。結果として1回の保存が「その日の既存行を全削除→新規行を全追記」という比較的重い同期になるが、他の記録と同じupsert/削除の仕組みをそのまま流用できる
3. 送信先
   - Cloudflare Workers(写真AI判定と同じ中継サーバー)を経由してGoogle Sheets APIを呼び出し、**アプリのデータモデル専用に新規作成したスプレッドシート**(今使っている手動運用の既存スプレッドシートとは別物)の7タブ(記録5タブ+マスタ2タブ)へ反映する
   - 反映は行末尾ID列をキーにした**upsert + 削除**で行う(Issue #30):
     - 新規(ID列に無い): `values.append` で末尾に追記
     - 編集(ID列に既に有る): `values.get` でID列を読んで行番号を特定し、`values:batchUpdate` で該当行を上書き(編集のたびに新しい行が増えて重複するのを防ぐ)
     - 削除(トゥームストーン): 同じくID列で行を特定し、`batchUpdate` の `deleteDimension` で行を物理削除する(複数行は行番号の降順で削除し、行番号ズレを避ける)。ID列に該当行が無ければ「削除済み」とみなして冪等に成功扱いにする
     - 体重記録タブ: `日付 | 体重(kg) | 体脂肪率(%) | メモ | 記録日時 | ID`(日付は`yyyy年mm月dd日`、記録日時は`yyyy年mm月dd日 hh:mm`の文字列としてJST表示に変換して書き込む)
     - 食事記録タブ: `日時 | 区分 | 料理名 | カロリー(kcal) | たんぱく質(g) | 脂質(g) | 炭水化物(g) | ID`(日時は同じく`yyyy年mm月dd日 hh:mm`のJST表示)
     - 水分記録タブ: `記録日時 | 摂取量(ml) | ID`(Issue #72)
     - 筋トレ記録タブ: `日付 | 種目名 | 種目順 | セット番号 | 重量(kg) | 回数 | 記録日時 | ID`(Issue #72)
     - 日記記録タブ: `日付 | 気分 | 本文 | 記録日時 | ID | 飲酒`(Issue #72。自由記述本文もそのままシートに書き出す。プライバシー面は認識した上で、バックアップの完全性を優先する判断とした)。飲酒列(F)は後付け(Issue #112)のため、既存シートのID列(E)を動かさないようID列の後ろに置く(種目マスタの部位列と同じ方針)。値は「あり」/空欄
     - 食事マスタタブ: `品目名 | カロリー(kcal) | たんぱく質(g) | 脂質(g) | 炭水化物(g) | 出典 | 登録日時 | ID`(Issue #96)
     - 種目マスタタブ: `種目名 | 登録日時 | ID | 部位`(Issue #96、部位はIssue #104)。部位列(D)は後付けのため、既存シートのID列(C)を動かさないようID列より後ろに置く。部位は日本語ラベル(胸/背中/...)で書き出し、取り込みは日本語ラベル・英語キーのどちらも受け付ける(空・未知の値は「分類なし」として行自体は取り込む)
     - 血圧記録タブ: `日付 | 最高血圧(mmHg) | 最低血圧(mmHg) | 脈拍(bpm) | メモ | 記録日時 | ID`(Issue #117)。IDは日付(体重・日記と同じ後勝ちの日付キー)
     - 周囲径記録タブ: `日付 | 腹囲(cm) | 胸囲(cm) | 太もも(cm) | メモ | 記録日時 | ID`(Issue #118)。IDは日付。胸囲・太ももは任意(空欄可)
     - 習慣マスタタブ: `習慣名 | 目標頻度(週) | アーカイブ | 並び順 | 登録日時 | ID`(Issue #113)。アーカイブ列は「アーカイブ」/空欄
     - 習慣記録タブ: `日付 | 習慣名 | 習慣ID | 記録日時 | ID`(Issue #113)。IDは`${日付}_${習慣ID}`の合成キー(1日1習慣1件の後勝ち)。習慣名はシートの可読性のための非正規化コピー
     - マスタ2タブ(食事・種目)およびフェーズ4の4タブ(血圧・周囲径・習慣マスタ・習慣記録)は後付けのため既存スプレッドシートには存在しない。同期時にタブが無ければ、Workerが見出し行付きで自動作成してから書き込む(フェーズ1〜2の記録5タブは手動作成が前提のまま)
     - Cloudflare WorkersはUTCで動くため、JST変換は`Intl.DateTimeFormat`に`timeZone: "Asia/Tokyo"`を明示して行う(`Date`の`getHours()`等をそのまま使うと9時間ズレる)
     - 既存の手動運用シートは「1日1行、その日の食事の合計カロリー/PFC」という集計形式でアプリの`MealRecord`(1食事=1レコード)とは粒度が異なるため、あえて合わせずに新規シートとした
   - クライアントに認証情報を直接持たせないよう、認証(Googleサービスアカウント)はWorkers側で保持する
4. 成功時
   - 送信(追記/上書き)を確定した記録の`synced`フラグを`true`に更新し、削除を確定した行の削除トゥームストーンを消す。最終同期日時(`lastSyncedAt`)を更新する
5. 失敗時(オフライン、APIエラーなど)
   - `synced`フラグは`false`のまま維持し、削除トゥームストーンも残す。記録自体は失わない
   - 次回のトリガー(アプリ起動時 or 手動)で再度同期を試みる
   - ユーザーには軽くエラーであることを伝える(派手なエラー画面は出さない)。設定画面のメッセージ表示以上の通知(バックオフ間隔・ホーム画面バッジ等)は追加しない(12章参照)

### 10.1 取り込み(スプレッドシート→アプリ。Issue #54)

スプレッドシートを復元可能なバックアップとして使うための、シート→アプリ方向の手動取り込み。機種変更・ブラウザデータ消失時の復元と、シートに手入力した過去データの移行に使う。

1. トリガー: 設定画面の「シートから取り込み(復元)」ボタンのみ(自動では実行しない)
2. 読み取り: クライアント(`runImport`)が `GET /api/import-sheets` を呼び、Workerが12タブ(体重・食事・水分・筋トレ・日記の記録5タブ+Garmin連携が書き込む「活動記録」タブ(#81)+食事マスタ・種目マスタの2タブ(#96)+血圧・周囲径・習慣マスタ・習慣記録の4タブ(#117・#118・#113))の全行を読み取ってレコード形式に逆変換して返す。活動記録タブはタブ自体が無ければ空として扱い(Garmin未連携でも取り込みは失敗しない)、マスタ2タブおよびフェーズ4の4タブも同様(初回同期前のシートには存在しないため)
   - 行の解析はトレラントに行う: 日付・日時はSheetsのロケール表示や手入力を考慮して複数表記(`yyyy年mm月dd日`・`yyyy/m/d`・`yyyy-m-d`、時刻省略可=JST 00:00扱い)を受け付け、数値は桁区切りカンマを許容する。解釈できない行はスキップして件数を返す(1行目は見出し行とみなしスキップ数に数えない)
   - 手入力行を想定し、体重の体脂肪率・メモ・記録日時、食事のPFC(省略時0)、水分・筋トレ・日記の記録日時(省略時はその日のJST 00:00)は省略可。食事の日時・区分・品名・カロリー、体重の日付・体重、水分の記録日時・摂取量、筋トレの日付・種目名・種目順・セット番号・重量・回数、日記の日付は必須(日記の気分・本文・飲酒は空を許容。飲酒列は「あり」のほかチェックボックス・手入力を想定した表記(TRUE・○・1)もtrueとして受け付け、それ以外は「記録なし」扱い。Issue #112)
   - マスタも同様に手入力行を想定する: 食事マスタは品目名・カロリーが必須でPFC(省略時0)・出典は省略可、種目マスタは種目名のみ必須。登録日時は省略時=取り込み時刻とする(マスタには代わりに使える日付キーが無いため)。同名(前後空白無視)の2行目以降は重複としてスキップする(#96)
   - **ID列が空の行にはWorkerがIDを採番し(体重=日付、食事・水分・筋トレ=UUID、日記=日付、食事マスタ・種目マスタ=UUID、血圧・周囲径=日付、習慣マスタ=UUID、習慣記録=`${日付}_${習慣ID}`の合成キー)、その場でシートに書き戻す**。書き戻さないと以後の同期(ID列キーのupsert・行削除)が該当行を見つけられず重複行を生むため、書き戻しに失敗した場合は取り込み全体を失敗させる。活動記録タブはID列を持たない(日付が主キーで、アプリ→シート方向の同期が無いため採番・書き戻しも不要)
3. マージ(クライアント側): **追加のみ**
   - ローカルに同じキー(体重・日記=日付、食事・水分・筋トレ・マスタ=ID)の記録が既にあればスキップする(ローカル優先。シート側の編集でローカルを上書きしない)
   - マスタはIDが違っても同名(前後空白無視)の既存品目・種目をスキップする — 手入力行はID採番前にローカルと同名になりえ、種目マスタは同名を許さない(サジェストのキーが名前)ため(#96)
   - 削除トゥームストーンが保留中のキーもスキップする(未送信の削除が取り込みで復活するのを防ぐ)
   - 取り込んだ記録は`synced: true`で保存する(シート由来のため再送信不要。シート側に同じIDの行が存在するので、取り込み後にアプリで編集すれば通常のupsertで正しく反映される)
   - 例外として活動記録は「追加のみ・ローカル優先」にせず、常にシート側の値で上書きする(アプリ内に編集・削除が無く競合しないうえ、Garminのバックフィルによる過去日の訂正を反映するため。#81)
4. 制約
   - シートに無い情報は復元できない: 食事のAI推定値・写真参照、設定(目標値)。完全バックアップが必要な場合はJSONエクスポート(`db/backup.ts`、UIなし)が該当(食事マスタ・種目マスタは#96で同期・取り込みの対象になった)
   - シートの日時は分単位のため、復元されたタイムスタンプの秒以下は失われる
